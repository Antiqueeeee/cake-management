### 1. 参与者词典

文中所有时序图复用以下参与者定义，避免每张图重复说明：

| 参与者 | 角色定位 | 备注 |
|--------|----------|------|
| 客户 | 在微信群/私聊预订蛋糕的顾客 | 系统不直接对接，通过店员转发消息 |
| 店员 | 接单与对话机器人交互的主角色 | 通过个人微信接入机器人 |
| 后厨 | 制作蛋糕、操作后厨屏的角色 | 物理位置在店里 |

| 系统 | 后端业务系统的统一抽象 | 内部按职责拆为订单服务、库存服务、客户档案、菜单、配方、鉴权等子模块，仅在需要展示内部分工时拆分 |
| 前端 | 浏览器侧 Web App（admin 后端管理页面 / kitchen 后厨屏） | v1.4 新增：登录、表单交互、调用 HTTP API |
| Caddy | 反向代理 + TLS 终止层 | v1.4 新增：所有外部入口（前端、OpenClaw）经 Caddy 进入 backend；只做 TLS 与转发、不做业务鉴权 |
| OpenClaw Plugin | OpenClaw 框架内置的 hook 机制 | v1.4 新增：在消息收发、工具调用等节点确定性触发回调，用于对话留痕；属 OpenClaw 进程内、不独立部署 |
| 后厨屏 | 后厨操作界面（平板 Web App） | 实时刷新订单、触发语音播报 |
| 运维群 | 店内全员微信群，最低保障通道 | 应急退阶时承载手动接单 |

### 2. Mermaid 语法约定

- `A ->> B`：实线箭头，请求/调用
- `B -->> A`：虚线箭头，返回/响应
- `A --) B`：异步通知（如新订单推送）
- `Note over A,B`：补充说明
- `alt / else / end`：条件分支
- `loop / end`：循环
- `opt / end`：可选分支

### 3. 通用约定（需求文档中的硬约束）

- **订单创建不扣库存**：原料扣减的唯一触发点是「成品诞生」（后厨备货入库或现做完工），订单创建、缺料检测均不动账面库存
- **缺料检测口径**：可用量 = 当前库存 − 所有「待制作+制作中」订单对该原料的需求总和（承诺量，仅覆盖现做路径）
- **状态约束**：仅「待制作」状态允许改单；其他状态需走「取消重下」
- **失败留痕**：取消、改单、入库、扣减等关键操作必须留痕，便于事后追溯

## 二、订单管理

订单管理是系统中枢。订单从客户消息进入系统到完成交付，经历 7 个状态（待制作、制作中、待取、待配送、配送中、已完成、已取消），任意状态均可取消但回滚策略不同。

### 故事 1：店员通过机器人创建新订单（happy path）

**对应需求**：第二章故事 1
**核心约束**：订单创建时不扣库存；缺料检测的承诺量覆盖所有「待制作+制作中」订单；后厨屏 2 秒内刷新+语音播报；支付状态在创建时由店员在消息中告知（已付则附带支付方式，未付则默认），机器人回执中显式展示供复核。

```mermaid
sequenceDiagram
    participant 客户
    participant 店员
    participant 机器人
    participant 档案 as 客户档案
    participant 菜单
    participant 配方
    participant 库存 as 原料库存
    participant 订单 as 订单服务
    participant 屏 as 后厨屏

    客户->>店员: 微信预订消息「明天下午3点要个蓝莓蛋糕」

    alt 客户已线上付款
        店员->>机器人: 转发「wxid_abc123 明天下午3点要个蓝莓蛋糕，已付 微信转账」
    else 客户未付款（默认）
        店员->>机器人: 转发「wxid_abc123 明天下午3点要个蓝莓蛋糕」
    end

    rect rgb(245, 245, 245)
    Note over 机器人,库存: 字段提取与缺料检测
    机器人->>档案: 按 wxid 查询历史
    档案-->>机器人: 电话138xx / 地址A小区 / 备注「偏爱紫色」
    机器人->>菜单: 匹配商品「蓝莓蛋糕」
    菜单-->>机器人: 命中，价格 38 元，保质期 2 天
    机器人->>配方: 读取蓝莓蛋糕配方
    配方-->>机器人: 淡奶油200ml / 蓝莓100g / 蛋糕胚1个
    机器人->>库存: 查询「需求量 / 可用量 / 承诺量」
    Note over 机器人,库存: 可用量=当前库存−Σ(待制作+制作中)订单需求
    库存-->>机器人: 各原料均充足（不缺料）
    end

    机器人-->>店员: 回执复核：客户/商品/时间/电话/地址/备注/支付状态/缺料检测，请求确认
    店员->>机器人: 回复「对」

    rect rgb(255, 250, 240)
    Note over 机器人,订单: 入库（不扣库存）
    机器人->>订单: 创建订单（字段已确认版本，含支付状态与支付方式）
    订单->>订单: 生成编号「0628-032」，状态置「待制作」
    Note over 订单: 此处不调用库存扣减
    订单-->>机器人: 返回订单编号
    end

    机器人-->>店员: 回执「已下单，编号 0628-032」
    店员->>客户: 转告订单编号

    订单--)屏: 推送新订单事件
    Note over 屏: 入库后 ≤2 秒内刷新+语音播报
    屏->>屏: 高亮新订单 + 播报「新单，蓝莓蛋糕，明天15:00取」
```

**验收要点**：
- 回执必须在 5 秒内返回
- 入库时不发生任何库存扣减动作
- 后厨屏刷新延迟 ≤2 秒
- 店员在创建消息中附带「已付 + 支付方式」时，回执必须显示支付状态、订单入库按已付写入；未附带时默认未付、支付方式为空

### 故事 2：消息缺信息时机器人追问

**对应需求**：第二章故事 2
**核心约束**：商品匹配不上时必须列出可选商品；wxid 不在档案时必须先收集电话/地址再下单。

```mermaid
sequenceDiagram
    participant 客户
    participant 店员
    participant 机器人
    participant 菜单
    participant 档案 as 客户档案
    participant 订单 as 订单服务
    participant 屏 as 后厨屏

    客户->>店员: 「要个蛋糕明天」
    店员->>机器人: 转发「wxid_new456 要个蛋糕明天」

    rect rgb(245, 245, 245)
    Note over 机器人,档案: 字段识别发现缺失
    机器人->>菜单: 模糊匹配「蛋糕」
    菜单-->>机器人: 匹配失败（菜单含蓝莓/芒果/...）
    机器人->>档案: 查询 wxid_new456
    档案-->>机器人: 无记录（新客户）
    end

    机器人-->>店员: 追问：「要哪个蛋糕？（菜单：蓝莓/芒果/...）；客户电话与地址是？」
    店员->>机器人: 补充「蓝莓，电话139xx，地址B小区」

    rect rgb(255, 250, 240)
    Note over 机器人,订单: 建档+建单（同一事务，仅指客户档案 wxid 创建与订单关联字段）<br/>不涉及订单备注与档案备注的双写场景（双写补救见客户档案故事3）
    机器人->>档案: 新建档案（wxid/电话/地址/创建时间）
    机器人->>菜单: 复核匹配「蓝莓蛋糕」
    菜单-->>机器人: 命中
    机器人->>订单: 创建订单
    订单-->>机器人: 返回订单编号
    end

    机器人-->>店员: 回执订单编号 + 已建档提示
    订单--)屏: 推送新订单（同故事1）
```

**验收要点**：
- 商品匹配失败时必须把可选菜单列出
- 新客户必须先建立档案（电话+地址）才能完成下单

### 故事 3：客户到店取货

**对应需求**：第二章故事 3
**核心约束**：按编号 1 秒内显示订单详情并展示当前支付状态；按钮可用性由支付状态决定——未付订单只能点「已付款 / 已交付」（合并操作，记录支付方式与收款时间戳），已付订单只能点「已交付」（仅切状态，支付状态保持不变）。

```mermaid
sequenceDiagram
    participant 客户
    participant 后厨
    participant 屏 as 后厨屏
    participant 订单 as 订单服务

    客户->>后厨: 报编号「我取 0628-032」
    后厨->>屏: 输入编号 0628-032

    Note over 屏,订单: 查询延迟 ≤1 秒
    屏->>订单: 按编号查询
    订单-->>屏: 返回订单详情（商品/数量/支付状态/客户标识）
    屏-->>后厨: 展示详情与按支付状态启用的按钮

    alt 子场景 A：取货时付款（订单未付）
        后厨-->>客户: 拿货、收款（线下完成）
        后厨->>屏: 点击「已付款 / 已交付」
        屏->>订单: 支付状态→「已付」+ 记录支付方式与收款时间戳<br/>+ 状态→「已完成」
        订单-->>屏: 确认变更
        屏-->>后厨: 显示已完成
    else 子场景 B：已付订单纯交付（订单已付，客户线上预付）
        后厨-->>客户: 拿货（无需再次收款）
        后厨->>屏: 点击「已交付」
        屏->>订单: 状态→「已完成」+ 记录交付时间戳<br/>支付状态保持「已付」不变
        订单-->>屏: 确认变更
        屏-->>后厨: 显示已完成
    end
```

**验收要点**：
- 订单详情必须显式展示当前支付状态（已付/未付），按钮可用性由支付状态决定，避免后厨误操作
- 子场景 A 状态变更必须同步落账收款时间戳与支付方式，不能事后补
- 子场景 B 只切订单状态，不得触发二次收款或支付状态变更
- 此流程仅适用于自取订单的「待取」状态

### 故事 4：客户取消订单（按订单状态分支）

**对应需求**：第二章故事 4
**核心约束**：取消必须留痕（含取消时状态、采用的回滚策略、操作人）；库存回滚策略由订单当前状态决定，配送中状态系统不自动调整库存。

```mermaid
sequenceDiagram
    participant 客户
    participant 店员
    participant 机器人
    participant 订单 as 订单服务
    participant 库存 as 库存服务
    participant 档案 as 客户档案

    客户->>店员: 告知取消
    店员->>机器人: 「取消 0628-032」
    机器人->>订单: 按编号定位订单，读取当前状态
    订单-->>机器人: 返回订单状态

    alt 订单处于「待制作」或「制作中」
        Note over 机器人: 成品未诞生，原料与成品库存账面均未变动
        Note over 机器人: 不扣减任何库存（制作完成前都不扣，统一原则）
        机器人-->>店员: 回执：「订单X状态，无库存变动；后厨实际领料差异通过盘点校正。确认取消？」
        店员->>机器人: 确认
        机器人->>订单: 状态置「已取消」，留痕

    else 订单处于「待取」或「待配送」
        Note over 机器人: 蛋糕已成型，原料已扣
        机器人-->>店员: 回执询问：「蛋糕处置方式？入成品库存 / 报损废弃」
        店员->>机器人: 选择「入成品库存」或「报损」

        opt 选择「入成品库存」
            机器人->>库存: 对应批次数量加回（成品库存）
        end
        opt 选择「报损废弃」
            机器人->>库存: 损耗记录追加一条
        end
        机器人->>订单: 状态置「已取消」，留痕处置结果

    else 订单处于「配送中」
        Note over 机器人: 蛋糕已交付配送员，物理上无法回滚
        机器人-->>店员: 回执：「配送中，系统不自动处理库存，请填协商结果」
        店员->>机器人: 填写协商结果（截回/折价/补做...）
        机器人->>订单: 状态置「已取消」，留痕协商结果（库存不动）
    end

    机器人-->>店员: 取消完成回执（含所采用策略与库存变动明细）
```

**特殊情况：配送员退回**（蛋糕实物回到店里）

```mermaid
sequenceDiagram
    participant 配送员
    participant 店员
    participant 机器人
    participant 订单 as 订单服务
    participant 库存 as 库存服务

    配送员->>店员: 退回蛋糕（拒收/地址错/联系不上）
    Note over 机器人: 系统不自动判定退回原因，由店员选择路径
    店员->>机器人: 报告退回 + 选择处置

    alt 选择「重新安排配送」
        机器人->>订单: 状态回到「待配送」
    else 选择「客户自取」
        机器人->>订单: 状态回到「待取」
    else 选择「入成品库存继续售卖」
        机器人->>库存: 按待取/待配送取消策略处理（入库或报损）
        机器人->>订单: 状态置「已取消」
    end
```

**验收要点**：
- 3 个状态分支的回滚策略必须严格按上图执行（待制作/制作中合并为同一种处理）
- 取消记录必须包含：取消时间、取消时状态、采用的策略、操作人
- 待制作/制作中取消时库存不得变动；后厨实际领料差异通过盘点校正，不在取消动作中处理
- 配送中取消时库存不得自动调整

### 故事 5：客户改单

**对应需求**：第二章故事 5
**核心约束**：仅「待制作」状态允许改单；商品/数量变化必须重做缺料检测；改单订单自身按新值计入承诺量；已付订单的价差不自动退补，仅提示店员与客户协商。

```mermaid
sequenceDiagram
    participant 客户
    participant 店员
    participant 机器人
    participant 订单 as 订单服务
    participant 菜单
    participant 配方
    participant 库存 as 原料库存

    客户->>店员: 「刚才那个 0628-032 改成芒果蛋糕」
    店员->>机器人: 转发「改 0628-032，改成芒果蛋糕」

   机器人->>订单: 按编号定位订单，读取状态
    订单-->>机器人: 返回订单状态与原字段

    alt 状态 ∈ {制作中, 待取, 待配送, 配送中, 已完成, 已取消}
        机器人-->>店员: 拒绝：「订单已进入制作/交付环节，无法修改，请取消重下或与客户协商」

## 七、成品库存

成品库存是「已经做好、但还没匹配订单的蛋糕」的缓冲池。

**核心设计取舍——原料扣减统一到「成品诞生」这一刻触发**（备货入库 + 现做完工两种来源，详见需求文档第七章第 1 节）：

1. **物理消耗与账面对齐**：原料物理消耗发生在做蛋糕的瞬间，账面应同步反映；订单创建只是预订归属，不扣料
2. **统一触发点 vs 路径分叉**：避免同时维护「订单创建扣料」与「成品诞生扣料」两条路径，订单创建只做预测性缺料检测
3. **现做订单的「形式化入库」**：现做完工时成品入库后立刻被本订单匹配扣减，表面经过库存、本质是「现做现领」，保持入库这一步是为了让原料扣减有统一触发点

本章 4 个故事中，故事 1（备货入库）与配方管理故事 2（现做完工）都包含「成品诞生」这一关键节点；故事 2（订单匹配扣减）展示订单如何消费已有批次（不再扣原料，避免双扣）。

## 九、接单机器人

### 故事 4：OpenClaw Bearer Token 鉴权

**对应需求**：第九章间接（鉴权属技术实现层，需求文档不写传输细节）
**对应技术方案**：§3.1 MCP 通信模式 + §5.1 鉴权双轨 + §7.2 部署拓扑
**核心约束**：Caddy 只做 TLS 终止、不做业务鉴权；Bearer Token 与 session cookie 是两套独立凭证（鉴权双轨）；token 写在 OpenClaw 配置文件中、不随工具调用频繁变化；token 轮换无需改动代码。

    Note over 店主,Token: 部署期一次性配置
    店主->>Token: 签发 token（INSERT api_token）
    Token-->>店主: 返回 token 明文（仅此一次可见）

    end

    OpenClaw->>OpenClaw: 启动 → 加载配置中的 Bearer Token
    Note over OpenClaw: openclaw mcp add --transport sse https://cake-shop.example.com/mcp/sse<br/>--header "Authorization: Bearer <token>"

    rect rgb(255, 250, 240)
    Note over OpenClaw,系统: 一次 MCP 工具调用
    OpenClaw->>Caddy: POST /mcp/sse<br/>Authorization: Bearer <token>

        OpenClaw->>OpenClaw: 重启或热加载配置
        Note over OpenClaw: 旧 token 立即失效、新 token 生效
    end
```

**验收要点**：
- Caddy 配置中不含任何业务鉴权逻辑（仅 TLS + 路径转发）
- token 校验失败时 OpenClaw 立即回执店员、不静默重试
- token 轮换流程不改 backend 代码、仅靠 `api_token` 表 + OpenClaw 配置变更
- Bearer Token 与 session cookie 两套凭证互不干扰（鉴权双轨，见 §5.1）

### 故事 5：后厨/店主 PIN 码登录

**对应需求**：第十章 §10.1 第 7 条（v0.8.3 新增）
**对应技术方案**：§6.2 PIN 码登录流程 + §5.2 鉴权模块 `/api/auth/*`
**核心约束**：admin（后端管理页面）与 kitchen（后厨屏）共用同一登录接口、仅 `client_type` 字段不同导致 session 有效期不同（admin=8h / kitchen=90d）；PIN 错 5 次锁 10 分钟；IP+用户名双维度限流 + 单 IP 每日上限 50 次兜底；session cookie HttpOnly + Secure + SameSite=Strict。

```mermaid
sequenceDiagram
    participant 用户 as 后厨/店主
    participant 前端 as 前端 Web App
    participant Caddy
    participant 系统 as backend (auth dependency + auth service)
    participant User as user 表

    用户->>前端: 打开浏览器加载 admin / kitchen
    前端->>前端: 检测无有效 session cookie
    前端-->>用户: 重定向到 /login

    rect rgb(255, 250, 240)
    Note over 用户,系统: 单 IP 每日上限 50 次兜底（先于 user 校验）
    用户->>前端: 输入 6 位 PIN（数字键盘）+ client_type
    前端->>Caddy: POST /api/auth/login<br/>{pin, client_type}
    Caddy->>系统: TLS 终止 + 反代（不做鉴权）
    系统->>系统: 检查该 IP 当日登录请求计数
    alt 当日 IP 计数 > 50
        系统-->>前端: 429 RATE_LIMITED（IP 兜底）
        Note over 系统: 不进 user 校验、直接拒绝该 IP 当日所有登录
    else 当日 IP 计数 ≤ 50
        系统->>User: 遍历 enabled=true 记录、对 pin_hash 用 bcrypt.checkpw
        alt locked_until > now（用户已被锁）
            User-->>系统: 用户处于锁定状态
            系统-->>前端: 429 RATE_LIMITED + detail 含剩余锁定时长
        else PIN 错误
            系统->>User: failed_attempts += 1
            alt failed_attempts 达到 5
                系统->>User: 置 locked_until = now + 10min、重置 failed_attempts = 0
            end
            系统-->>前端: 401 UNAUTHORIZED + AUTH_INVALID_PIN
        else PIN 正确
            系统->>User: 重置 failed_attempts、locked_until = NULL、更新 last_login_at
            系统->>系统: 签发 session cookie<br/>（HttpOnly + Secure + SameSite=Strict）<br/>有效期：client_type=admin → 8h<br/>client_type=kitchen → 90d
            系统-->>前端: 200 + Set-Cookie + user info（display_name / role）
            前端-->>用户: 进入主界面（操作人即登录用户）
        end
    end

    rect rgb(245, 245, 245)
    Note over 用户,系统: 后续业务请求
    用户->>前端: 操作（如改单、查库存）
    前端->>Caddy: 业务请求带 session cookie
    Caddy->>系统: 反代
    系统->>系统: auth dependency 校验 session cookie
    系统-->>前端: 业务响应
    end
```

**验收要点**：
- admin 与 kitchen 共用 `/api/auth/login` 端点、仅 `client_type` 字段决定 session 有效期（admin 8h / kitchen 90d）
- PIN 错 5 次锁定 10 分钟、返回 429 + 剩余时长；锁定期间同 PIN 再试直接 429
- 单 IP 每日上限 50 次兜底先于 user 校验、防「同 IP 对不同用户各试 5 次」的缝隙
- session cookie 三道保护：HttpOnly（防 JS 读）+ Secure（仅 HTTPS）+ SameSite=Strict（防 CSRF）
- 防爆破：浏览器路径锁 10 分钟后 6 位 PIN 需 ~3.8 年穷举

## 十一、应急退阶策略

系统以「运维微信群」作为最低保障通道，任何环节挂掉都退回到「店员在运维群发订单信息 + 后厨看群接单」的手动流程。所有场景恢复后都通过「补录流程」把手工期间的订单回流到系统。

> **补录流程通用约定**（适用于所有场景的恢复阶段）：
> 1. 编号按订单实际下单日期当日位置生成（不取补录日期）
> 2. 若店员在运维群期间已手工分配编号，补录优先采用（除非已被系统占用）
> 3. 下单时间取运维群消息时间或纸笔记录时间（不取补录时刻）
