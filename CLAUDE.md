## 开发前置：场景澄清（BDD 思维）

任何功能在动手写代码之前，必须先弄清楚"这个功能在真实场景里是怎么被用的"。没有场景就没有验收标准，没有验收标准写出来的代码就是脑补。

### 必须回答的问题

- **谁用？** 顾客 / 员工 / 后厨 / 老板 / 管理员 / 系统自动调度
- **什么时候用？** 触发场景，比如"库存低于安全线时"、"顾客在小程序里点 DIY 蛋糕时"
- **怎么用？** 输入是什么、期望输出是什么、关键交互路径
- **不用会怎样？** 不属于这个功能边界的情况，明确排除
- **做完什么样？** 可观测的验收标准（不是"实现完毕"，而是"用户能完成 X 操作并看到 Y 结果"）

### 产出物（轻量，不要写成正式文档）

- 一句话用户故事："作为 X，我希望 Y，以便 Z"
- 关键路径列表：happy path + 2~3 个主要边界
- 验收标准：能被 Playwright 脚本验证的、用户视角的行为

### 红线

- 场景不清楚不开工，先问业务方 / 产品 / 用户，不脑补
- 验收标准不能是"功能实现了"，必须是"用户能完成某事"
- 边界情况和异常路径在开工前列出来，不要写完再补

## 真实验证准则

完成代码 ≠ 完成任务。适用于所有项目，不可跳过。

### 验证分层（按项目实际情况执行可用层）

- **单元测试**：验证逻辑正确性，但自己写的测试只能验证实现，不能验证意图
- **集成测试**：验证模块间通信、API 端点、数据库操作。有后端的项目必须有
- **真实运行验证**：后端用 curl/httpie 调真实接口确认响应；前端用 Playwright 或浏览器工具打开页面确认渲染和交互；CLI 工具直接执行命令确认输出。不能只在脑子里"觉得应该没问题"
- **E2E 测试**：**每个用户可见的功能都必须用 Playwright 等工具从用户场景角度编写自动化脚本，在真实浏览器中跑通。脚本要面向[开发前置]里定义的场景，跑通后保留作为回归测试资产，不是这次跑过就丢。**

### 行为红线

- 禁止"写完单测全绿就宣布完成"——单测通过只是第一关，不是终点
- 禁止自己写代码又自己写测试就当验证过了——验证必须包含至少一个"代码之外"的检查（启动服务、curl 接口、浏览器访问）
- 改动涉及前后端交互时，必须验证真实请求链路，不能只 mock
- 禁止"功能开发完不写 E2E"——只要功能涉及页面交互，必须有用 Playwright 等工具编写的、面向使用场景的脚本，并跑通后才能声称完成
- 声称"已完成"之前，必须贴出真实运行的证据（命令输出、截图、或测试结果）

### 验证顺序

1. 跑单元测试 → 2. 跑集成 / E2E 测试（如有）→ 3. 启动服务做真实验证 → 4. 贴出证据 → 5. 才能说"完成"

### Fail loud：禁止隐性失败

任何"完成"声明都必须显式列出三类信息，缺一不可：

- **跳过了什么**：哪些用例 / 数据 / 分支被 skip 或被过滤掉
- **有什么 warning**：编译告警、运行告警、迁移过程中触发的约束失败
- **没验证到什么**：哪些路径 / 边界没跑过、用户提到的某个 case 是否真的测了

"迁移成功 / 测试通过 / 功能 OK"这种总结句默认不可信，必须伴随上述清单。
最贵的失败是看起来像成功的失败——宁可显式承认"这块我没确认"，也不要让用户隔了好几天才发现数据被悄悄跳过 / 某个 case 没测到。

### 表达清晰：面向所有读者

回复问题和生成 commit 消息时，默认读者并不一定看过项目代码。表述必须提供必要背景、说明问题影响和改动意图，让没有上下文的人也能理解，而不是只写给已经熟悉代码的人看。

### 解决方案原则：面向根因和项目长期质量

除非用户明确要求临时打补丁，否则方案应从技术专家视角出发，优先定位并解决根本原因。解决问题时要面向项目整体质量、架构一致性和长期可维护性，而不是只针对当前现象做局部绕过。

## 提交后审查：子代理 review

每次 commit 之后，必须启动一个空上下文的子 agent 来 review 本次改动。子 agent 不共享当前对话历史，只看到本次 diff 和主 agent 主动提供给它的上下文——这是有意为之，避免它被"我自己刚写完所以觉得没问题"的惯性带偏。

### 子 agent 的输入（由主 agent 整理后传入）

- 本次 commit 的完整 diff
- 这个改动对应的需求 / 场景描述（来自[开发前置]阶段）
- 必要的项目背景（架构约定、相关模块说明）
- 明确的 review 重点

不能让子 agent 自己猜上下文，主 agent 必须把"它在审什么、为什么这么改"讲清楚。

### 子 agent 的职责

- **合理性**：改动是否符合本次需求？是否解决了真问题？有没有跑偏？
- **正确性**：有没有 bug？边界条件、异常路径、并发情况是否覆盖？
- **范围**：有没有不该改的文件被改了？有没有"顺手优化"邻近代码？
- **一致性**：是否符合架构约定？有没有引入第二套互相冲突的模式？
- **可读性**：命名、结构、注释是否清晰？（但不纠结风格细节）

### 主 agent 收到 review 后的义务

- 显式回应每一条意见：同意 / 不同意 + 理由
- 同意的立刻修，不同意的说明依据，不能默不作声
- 修改后再跑一次相关验证，不能"改完就算"

### 红线

- 不能跳过 review 流程，哪怕改动很小
- 不能"我自己看了一遍觉得没问题"代替子 agent review——自己 review 自己等于没 review
- review 发现的问题如果选择不修，必须在交付说明里显式列出，不能闷掉

## 代码撰写八荣八耻

荣：明白接口来自哪、为何如此，必要时把上下游依赖全查清；
耻：不看文档、不问清需求、靠自己脑补去"猜"接口。

荣：每个操作前明确目标、边界条件、输入输出；
耻：带着模糊理解就动手写，写完再靠祈祷通过。

荣：对不懂的业务逻辑直接问业务方、产品、后端，不装懂、不假设；
耻：靠臆测补齐缺失信息，把自己的误解当成事实。

荣：优先复用现有接口、组件、规范，让系统保持一致与可维护；
耻：为图爽随手造一个新接口、新工具、新方法，埋雷给后来人。

荣：写完代码必须验证——包含边界、异常、并发、性能路径；
耻：只测自己最顺的那条 happy path，剩下交给运气和线上用户。

荣：代码结构服从架构，风格遵循团队规范，哪怕重构成本高；
耻：写完管他三七二十一，能跑就行，把架构当摆设。

荣：承认"不知道"并寻求帮助，用事实和数据做决策；
耻：硬撑、乱解释、瞎拍胸脯，用自信掩盖无知。

荣：修改代码前先理解原意、评估影响、完整验证，再动刀；
耻：看到问题就急着改，结果把一处 bug 修成三处事故。

荣：用最少代码解决问题，复用优于发明，简单优于"灵活"；
耻：写 200 行能用 50 行搞定的，硬塞用户没要求的"配置项""扩展点""未来兼容"。

荣：外科手术式改动，每行修改都能追溯到本次需求，不要清理别人的"屋子"；
耻：顺手"优化"邻近代码、改格式、删自己看不顺眼的旧实现，让 diff 变成考古现场。

荣：发现 codebase 里有两套互相冲突的模式时，挑一个（更新 / 更核心 / 更被依赖那套）继续写，并显式说明为什么挑它，把另一套标记为待清理；
耻：写出"两边都满足"的折中代码——双重 error handler、新旧 API 同时调、两套状态管理混用——折中产物比任何一种纯种实现都更难维护，也最难追 bug。

## 文档维护原则

文档不是越多越好。什么都记等于什么都没记——后来人面对一堆过期、互相矛盾的文档，比完全没有文档还糟。

### 少而精

- 每个文档必须有明确的用途和读者，没有的就删
- 不为"completeness"写文档，只为"有人需要看"写文档
- 同一信息只在**一个地方**维护（单一可信源），其他地方只放链接

### 随改动同步

- 开发过程中如果变更了最初的设计、接口、字段、流程，**必须同步更新对应文档**，作为本次改动的一部分，不另起任务
- 文档和代码不一致比没有文档还危险——做完功能后留下的旧文档会误导后来人
- commit 里包含行为变更时，diff 里应当能看到对应文档的更新；如果没有，要么文档不需要改（说明原因），要么就是漏了

### 过期就处理

- 无法继续维护的文档，明确标"已废弃"或直接删除
- 半真半假的文档最危险，宁缺毋滥

### 文档只写"为什么"和"怎么用"

- 代码本身已经表达了"是什么"，文档不重复
- 文档写：决策的原因、上下的约束、使用的步骤、踩过的坑
- 不写：字段清单、API 签名、函数签名——这些代码会变，文档会过期
