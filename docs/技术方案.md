## 一、技术栈选型

### 1. 选型总览

| 维度 | 选定 | 关键理由 |
|---|---|---|
| 后端语言 | Python 3.11+ | 用户指定；生态成熟 |
| 后端框架 | FastAPI | API-first、自动 OpenAPI 文档、类型注解自动校验请求体 |
| ORM | SQLAlchemy 2.0 | Python ORM 事实标准 |
| 数据库迁移 | Alembic | SQLAlchemy 配套、版本化迁移、支持回滚 |
| 数据库 | PostgreSQL 15 | JSONB 适合存留痕/快照、并发性能、事务隔离 |
| 前端框架 | Vue 3 + Composition API | 用户选择 |
| UI 组件库 | Element Plus | 用户选择；国内主流、中文文档 |
| 前端构建 | Vite | Vue 官方推荐、HMR 快 |
| 前端状态 | Pinia | Vue 官方推荐 |
| 前端语言 | TypeScript | 类型安全；后端 OpenAPI 可自动生成 TS 类型 |
| 机器人接入 | OpenClaw + MCP server | OpenClaw 原生支持 MCP、工具协议标准化 |
| 反向代理 | Caddy 2 | 自动 TLS（Let's Encrypt 自动签发与续签）、配置极简（Caddyfile 十行级别）、单二进制部署；统一承担 HTTPS 终止、端口屏蔽（backend 不直接对外）、路径过滤（仅放行 `/api/*`、`/mcp/*`、`/health/*` 与前端静态资源）；备案期无公网域名时自动降级为内部自签证书 |
| 鉴权机制 | session cookie（前端）+ Bearer Token（OpenClaw） | 前端走 PIN 码登录获取 session cookie；OpenClaw 持长期 Bearer Token 调用 MCP |
| 容器化 | Docker + docker-compose | 单机部署足够、K8s 过度设计 |
| CI/CD | GitHub Actions | 项目托管在 GitHub |

### 2. 关键决策的取舍

#### 为什么 FastAPI 不是 Django

业务核心是「机器人 MCP server + 后厨屏 HTTP API + 后端管理页面 HTTP API」三类 API，FastAPI 的 API-first 模型与业务高度契合。FastAPI 自动生成的 OpenAPI schema 可直接给前端 TypeScript 类型代码生成（`openapi-typescript`）使用，前后端类型一致。Django 自带 admin 看似省事，但本项目的「后端管理页面」要做的不止是 CRUD（订单状态机、客户档案异步补偿、机器人功能镜像等业务逻辑），Django admin 适配复杂业务反而更累。

代价：FastAPI 不带 admin，店主后台维护菜单/原料/配方等 CRUD 需要自己写。但 Vue 3 + Element Plus 的 Table + Form 组件已经把这些模式化了，开发成本可控。

#### 为什么 PostgreSQL 不是 SQLite

PostgreSQL 的 JSONB 字段适合存储「订单留痕」「配方快照」「操作日志」等半结构化数据；并发性能在「成品诞生扣料 + 订单匹配扣减」这种事务场景下更可靠；后续第二期补库存管理时，行级锁、事务隔离级别等高级特性会有用。SQLite 虽然轻量但并发写入受限，docker-compose 部署 Postgres 是一行配置的事。

#### 为什么 SQLAlchemy + Alembic 不是裸 SQL

需求会迭代，数据库结构会变（第二期补成品库存/现货销售/统计都会新增表），版本化迁移是必需的。Alembic 把每次表结构变更记录为版本文件，支持升级/回滚，避免手工维护 SQL 脚本。SQLAlchemy 2.0 的类型注解 ORM 也与 FastAPI 的类型系统对齐。

#### 为什么 MCP server 不是直接 HTTP API

OpenClaw 原生支持 MCP（Model Context Protocol）作为工具调用协议，MCP 提供标准化的工具描述、参数校验、调用统计能力。把后端业务能力封装为 MCP server，机器人侧无需关心底层 HTTP 细节，工具的语义描述由后端统一维护。如果直接给机器人暴露 HTTP API，机器人需要自己处理参数拼装、错误重试等逻辑，不如 MCP 优雅。

代价：MCP server 是新增一层封装，但这一层薄（直接调用后端 service 层），实现成本低。

#### 为什么 monorepo 不是前后端分仓库

单人开发、前后端强相关，跨前后端的改动（新增 API + 前端调用）一个 commit 完成、Issue/PR 历史集中、共享类型定义天然在一起。monorepo 的复杂度（依赖管理、增量构建）只对大团队才是问题，本项目用不上。

#### 为什么 Caddy 不是 Nginx

本项目部署在云上单台服务器、对外提供 HTTPS 服务（架构 A：全云部署，详见第二章 §4），有公网 IP 但域名备案需要 1-3 周。Caddy 的「自动 HTTPS」能力是核心优势：备案完成后配置域名即可自动从 Let's Encrypt 签发证书并自动续签，零运维；备案期间（无可用域名）Caddy 也能用内部自签证书兜底（仅作为过渡，详见第七章 §2）。Caddyfile 十行级别即可声明 HTTPS + 反代 + 路径过滤，配置学习成本几乎为零；Nginx 配置 HTTPS + 反代 + 自动续签至少几十行 + 多个 server block + 单独的 certbot 维护流程，对单人维护的小店场景是过度复杂。**注意 Caddy 不承担业务鉴权**——session cookie 与 Bearer Token 校验由后端 FastAPI dependency 完成（详见第五章 §1），Caddy 只在传输层做 TLS 终止、端口屏蔽、路径过滤；这避免在 Caddy 里维护一套与后端重复的鉴权规则。

代价：Caddy 生态比 Nginx 小、性能在极限场景下略低，但本项目流量极低（单店每天几十单），Caddy 性能完全够用。

#### 为什么 session cookie + Bearer Token 双轨而不是统一一种

前端（后厨屏、后端管理页面）由人在浏览器操作，PIN 码登录获取 session cookie 是最自然的鉴权方式（浏览器自动管理 cookie、无需前端额外处理）。OpenClaw 是无人值守的程序，没有「人在浏览器输入 PIN」的环节，必须用长期 token；且 OpenClaw 未来可能从同机迁移到独立主机，Bearer Token 在 HTTP header 中携带、与传输层解耦，迁移时仅改 URL 与 token 即可。

代价：两套鉴权机制需要后端鉴权中间件同时支持两种凭证来源。但实现成本可控（FastAPI dependency 中区分两种凭证来源即可），换来的是前端用户体验与 OpenClaw 部署灵活性。

关键边界：
- **架构 A（全云部署）**：backend / DB / Caddy / OpenClaw 全部部署在云上单台服务器；店内仅有客户端设备（后厨屏平板、店主/店员电脑或手机、店员用来跟 OpenClaw 对话的微信客户端）；客户端通过公网 HTTPS 访问云上 Caddy
- 三个前端入口（OpenClaw、后端管理页面、后厨屏）**全部经过 Caddy 反向代理**，Caddy 统一做 TLS 终止（Let's Encrypt 自动证书）、端口屏蔽（backend 8000 仅容器内可达）与路径过滤（仅放行 `/api/*`、`/mcp/*`、`/health/*` 与前端静态资源，其他 404）；**业务鉴权（session cookie / Bearer Token）由后端 FastAPI dependency 承担、不在 Caddy 配置**，避免双重维护；Caddy 后端通过容器内部 HTTP 转发到 backend 服务，backend 服务**不直接对外暴露端口**
- OpenClaw 与 backend 同在云上，但**仍通过公网 HTTPS（Caddy 反代）调用 MCP server**（HTTP SSE 传输）——OpenClaw 是 OpenClaw 框架自身的进程、不与 backend 共进程，仍走标准网络调用；OpenClaw 持长期 Bearer Token
- 前端用户 PIN 码登录后持 session cookie 经 HTTPS 调 HTTP API（鉴权机制细节见第五章 §1）
- 三个前端入口共用同一套后端业务 service，确保业务规则一致
- 后端管理页面与机器人共用同一套 HTTP API + MCP 工具，互为「机器人功能镜像」（详见需求文档第一章第 3 节支撑组件 3）
- 后厨屏通过 WebSocket 接收新订单实时推送 + 提示音播报触发
- **架构 A 的固有弱点**：店内断网即全系统停摆（详见第七章 §2 末「架构 A 风险与缓解」与第八章 §7 平板保障）

### 3. 关键代码组织约束（避免「大文件」「耦合」痛点）

针对开发期两个常见痛点——前端单文件大几千行、后端单文件多功能耦合——本方案设定以下约束：

#### 前端约束

1. **每个 view 是独立文件**：一个页面一个 .vue 文件，单文件超过约 300 行触发拆分（拆出 components 或 composables）。阈值是经验值，目的是避免单文件失控
2. **每个模块自包含**：模块目录下含 views/components/api/store，模块间不直接互调 store，必须走 api 层
3. **API 调用集中在 api/ 目录**：view 不直接发请求，统一通过 `modules/xxx/api/` 调用，便于 mock 和重构
4. **共享代码放 shared/**：跨模块复用的组件、工具、类型放 `src/shared/`，模块内自用的不放

#### 后端约束

1. **router 是薄层**：只做参数校验、调用 service、返回响应，禁止写业务逻辑
2. **service 是业务核心**：所有业务逻辑（事务、状态机变更、库存联动、留痕）在 service 层，事务边界在 service 层
3. **禁止跨模块直接互调 ORM**：模块 A 不能直接 import 模块 B 的 SQLAlchemy 模型；必须通过模块 B 的 service 层方法（避免数据耦合）
4. **MCP 工具是 service 的薄包装**：MCP 工具不写业务逻辑，直接调用对应 service 方法
5. **数据库变更必须走 Alembic 迁移**：禁止手工改表结构，所有变更通过 `alembic revision` 生成版本文件
6. **模块内文件粒度按需细化（触发式拆分）**：默认每个模块用 `router.py / service.py / models.py / schemas.py` 四文件扁平结构；当**单个 `service.py` 实际行数 ≥ 300 行（含注释与空行，开发者凭直觉判断即可、不需精确计数）**时再拆为子目录（`services/` 按业务行为 create/update/lifecycle/cancel/query 拆、`models/` 按表拆、`routers/` 与 `schemas/` 同步拆、加 `constants.py` 放枚举与错误码），见 §2.2 order 模块示例。**「先扁平、膨胀了再分」比「一开始就分好」约束力更强**——因为分是有触发条件的、不是教条的，避免简单模块被强行套复杂结构。阈值 300 行是经验值，目的是避免单文件失控；判定基于实际行数而非预期

### 4. 部署拓扑

**架构 A（全云部署）**：backend / DB / OpenClaw / Caddy 全部部署在云上单台服务器（业主已在云上跑 OpenClaw、店内无公网 IP）。店内仅有客户端设备——后厨屏平板、店主/店员电脑或手机浏览器、店员用来与 OpenClaw 对话的微信——通过公网 HTTPS 访问云上服务。

容器组成（全部在云上单台服务器）：
- `caddy` 容器：Caddy 2 反向代理，对外暴露 443 端口（HTTPS）与 80 端口（仅做 301 跳转到 443 + ACME HTTP-01 挑战用），统一做 TLS 终止（Let's Encrypt 自动签发与续签）、端口屏蔽与路径过滤（业务鉴权由 backend 承担，不在 Caddy 配置）；通过容器内部网络转发到 backend:8000
- `backend` 容器：FastAPI + Uvicorn，**仅容器网络内可达，8000 端口不直接对外**；serve 前端构建产物（单镜像部署，简化）
- `postgres` 容器：PostgreSQL 15，**仅容器网络内可达**，数据卷持久化
- OpenClaw 与上述容器同在云上服务器，但**作为独立进程运行**（不进 docker-compose、不与 backend 共进程），通过公网 HTTPS（经 Caddy 反代）调用 MCP 端点；OpenClaw 自己的进程管理由业主已有的部署方式承担（systemd / supervisor / 手动等，不在本项目范围内）

**为什么定架构 A 而不是店内本地服务器**：业主澄清 OpenClaw 已经在云上跑了一段时间、店内只有客户端设备而无服务器、也无公网 IP。架构 A 与现状对齐，无需新增店内服务器硬件投入；公网可达性也让店主在外（出差、不在店里）通过手机浏览器查单成为可能，比店内本地服务器更灵活。

**架构 A 风险与缓解**：
- **店内断网即停摆**：架构 A 下，店内 WiFi/宽带断了，后厨屏连不上 Caddy、新订单无法接、状态无法推进，整个系统不可用。需求文档第十一章 §3「断电/断网应急策略」中「店内服务器可用、仅机器人挂」的前提不再成立——架构 A 下任何公网链路故障都等同于系统级不可用。**缓解**：(a) 业主确保店内宽带稳定性（建议拉两条不同运营商宽带做主备，或在手机热点作为应急网络）；(b) 长期评估方案——若业主后续愿意购置店内小型服务器，可平移到「店内主、云上备」的混合架构（即 backend/DB 下沉店内、OpenClaw 留云上），当前不实施；(c) 短期退阶仍走需求文档第十一章场景 1——店员用微信群人工接单、事后补录
- **公网暴露面增大**：架构 A 下 backend 经 Caddy 暴露到公网，攻击面比店内局域网大；缓解靠第六章 §4 安全机制（HTTPS + PIN + Bearer Token + 限流）+ Caddy 仅放行 `/api/*`、`/mcp/*`、`/health/*`、前端静态资源路径，其他 404

### 5. 端到端数据流（订单创建为例）

1. 客户在微信群发预订消息
2. 店员复制消息 + 客户 wxid，发给云上的 OpenClaw
3. OpenClaw 持 Bearer Token 经公网 HTTPS（Caddy 反代）调用 MCP 工具 `parse_order_message`（解析订单字段）
4. OpenClaw 调用 MCP 工具 `query_customer_archive`（查档案）
5. OpenClaw 调用 MCP 工具 `check_inventory_shortage`（缺料检测）
6. OpenClaw 整理回执发给店员确认
7. 店员确认后，OpenClaw 调用 MCP 工具 `create_order`（创建订单）
8. MCP server 调用 `order.service.create_order`，写入 PostgreSQL
9. service 层通过 WebSocket（WSS 经 Caddy）广播「新订单」事件给后厨屏
10. 后厨屏实时刷新 + 触发提示音播报

## 三、机器人接入方案（MCP server）

### 1. MCP server 在系统中的位置

MCP server 是后端 FastAPI 应用的一部分（同进程、共用 SQLAlchemy session、共用业务 service 层），通过 **HTTP SSE**（Server-Sent Events）与 OpenClaw 通信。OpenClaw 通过 MCP 协议发现工具列表、调用工具、接收结果。

**为什么定 HTTP SSE 而不是 stdio**：stdio 模式要求 OpenClaw 与 backend 同进程或同机启动；本项目架构 A 下 OpenClaw 与 backend 虽同在云上服务器、但作为独立进程运行（不进 docker-compose、不共进程），HTTP SSE 走 HTTP 协议是唯一可用的传输方式。HTTP SSE 也保留了未来把 OpenClaw 平移到独立主机的灵活性（仅改 URL 与 token 配置，不动 backend 代码）。注意「同进程、共用 SQLAlchemy session、共用业务 service 层」是指 **backend 进程内部**的 MCP server 子应用与业务 service / ORM 同进程——Caddy 反代只是 OpenClaw 与 backend 之间的传输层，backend 内部「MCP server ↔ service ↔ ORM 同进程共用 session」的进程模型完全不变。

**鉴权机制**：OpenClaw 在配置中持有一个长期 Bearer Token，所有 MCP 工具调用经 Caddy 反代后由 backend FastAPI 鉴权 dependency 校验（Caddy 只做 TLS 与转发、不参与鉴权，详见第五章 §1 鉴权双轨）。Token 写在 OpenClaw 配置文件中，不随工具调用频繁变化；token 泄露时可在后端 `api_token` 表中删除旧记录、签发新 token 并同步更新 OpenClaw 配置，无需改动代码。

技术实现：用 MCP Python SDK（`mcp` 包）实现工具定义并以 HTTP SSE 模式暴露端点（如 `/mcp/sse`），OpenClaw 通过 `openclaw mcp add --transport sse https://cake-shop.example.com/mcp/sse --header "Authorization: Bearer <token>"` 命令注册本系统的 MCP server，工具列表自动同步。

### 3. 错误降级

MCP server 调用后端 service 失败时的处理：

1. **瞬时错误**（DB 锁、网络抖动）：MCP server 内部重试 3 次，仍失败则返回错误结构
2. **持久错误**（service 抛业务异常）：MCP server 返回错误结构，OpenClaw 据此回执店员
3. **后端服务整体故障**：OpenClaw 检测不到 MCP server，回执店员「系统暂时不可用，请在运维群手动接单」（对接需求文档第十一章场景 3）

机器人侧硬故障（OpenClaw 本身挂）走需求文档第十一章场景 1：店员通过后端管理页面手动操作（首选）或运维群（兜底）。

### 4. OpenClaw Plugin 编排（v1.0.6 新增，bot_conversation_log 数据来源）

需求文档第九章 §4-(5) 要求留痕机器人接单全过程五类对话痕迹（店员原始消息 / 识别结果 / 机器人回执 / 店员回复 / 最终入库订单）。v1.0.5 原方案是「让 OpenClaw 在 LLM prompt 中被要求主动调用 `log_conversation_step` MCP 工具」——但这依赖 LLM 自觉、不可靠（LLM 可能漏调、跳过、被剪枝），且要求每条对话都多一次 MCP 调用增加延迟。

**v1.0.6 改为 Plugin hook 方案**：编写一个轻量 OpenClaw Plugin（TypeScript 实现、约 50-80 行），通过 Plugin SDK `api.on(...)` 注册三个 hook：

| Hook | 触发时机 | 写入 bot_conversation_log 哪一类 |
|---|---|---|
| `message_received` | OpenClaw 收到店员发来的微信消息 | 「店员原始消息」+「店员回复」（按 session 内是否已有记录区分首次/后续）|
| `message_sent` | OpenClaw 把回执投递回店员微信 | 「机器人回执」 |
| `before_tool_call`（仅监听 create_order 工具）| LLM 决定调用 create_order 时 | 「识别结果」（payload 为 LLM 解析出的订单字段）|

「最终入库订单」由 MCP 工具 `create_order` 自身在 service 层事务中写入（不依赖 Plugin），与 v1.0.5 一致。

每个 hook 把捕获的事件 POST 到后端新增的内部端点 `POST /api/internal/bot-conversation-log`（详见第五章 §2 内部端点说明），后端校验 Bearer Token 后写入 `bot_conversation_log` 表（结构见第四章 §2）。

**为什么 Plugin hook 优于 LLM 自觉调工具**：
- **确定性触发**：hook 在每条消息/每次工具调用时由 OpenClaw 框架确定性触发，不依赖 LLM 是否记得调；漏写风险从「LLM 行为不确定」降到「Plugin 进程挂掉」（后者有进程级监控）
- **无额外延迟**：hook fire-and-forget 异步 POST 到后端，不阻塞 LLM 推理；与「LLM 多调一次 log_conversation_step 工具」相比每条对话省一次模型往返
- **Plugin SDK 文档参考**：[OpenClaw Plugin hooks · OpenClaw Docs](https://docs.openclaw.ai/plugins/hooks) 列出 `message_received` / `message_sent` / `before_tool_call` 均为受支持的 observation hook，且这三个 hook 不需要 `allowConversationAccess: true` 高权限配置

**实现约束**：
- Plugin 由本项目维护、随 OpenClaw 配置一起部署（不发布到 ClawHub 等公共市场）
- Plugin 内部对每个 hook handler 加 try/catch 与 5s 超时，hook 失败仅写 OpenClaw 侧 stderr 日志、不抛异常（避免影响 OpenClaw 主流程，符合 Plugin SDK 文档的「Handler failures are logged and do not change the delivery result」原则）
- session_id 由 Plugin 从 `ctx.sessionKey` 派生（OpenClaw 框架为每个接单会话分配稳定 sessionKey）；同一 sessionKey 的多条消息归到同一 session_id
- 后端 `/api/internal/bot-conversation-log` 端点仅在 `api_token` 表中校验 Plugin 持有的 token（与 OpenClaw 调 MCP 的 token 同源或单独签发均可，由部署时决定；建议同源以减少 token 管理面）

**Plugin 失败的降级**：Plugin 进程挂掉时（OpenClaw 重启、Plugin 代码 bug）会丢失挂掉期间的对话痕迹；但「最终入库订单」仍由 service 层确定性写入 `order` 表，订单本身不会丢。店主发现 `bot_conversation_log` 缺失某段时，可对照 `order_status_log` 与 OpenClaw 自身的 LLM 调用日志（OpenClaw 自带）补全追溯——业务留痕以后端为单一可信源、OpenClaw 自带日志作调试补充，二者并存不冲突。

### 2. 关键表清单（仅列约束，字段细节由 SQLAlchemy 模型定义承担）

按业务模块组织，每张表的字段细节在 `backend/app/modules/xxx/models.py` 中定义。本节只列关键约束：

#### 菜单模块
- `menu`：菜名唯一约束；含价格、保质期、是否定制、是否在售字段
- `menu_global_config`：全局临期阈值比例

#### 原料模块
- `ingredient`：原料名唯一约束；当前库存数量允许为负
- `ingredient_stock_log`：每次库存变动留痕；第一期来源含「采购入库 / 成品诞生扣减 / 报损 / 盘点校正」四类（**「成品诞生扣减」在第一期由 `finish_cooking` 按配方扣一步完成**；第二期补成品库存后该来源拆为「备货入库 + 订单匹配扣减」两步，本表「来源类型」字段取值集合扩展，但表结构不变，详见需求文档第十三章第 4 节取舍 1）；关联订单或采购单（可空）；必填原因 + 操作人 + 时间
  - 注意：**订单取消不写本表**（改单仅允许在「待制作」状态，本身不涉及库存变动，也不写本表）。原料消耗在「成品诞生」时已扣且不可回滚（需求文档第五章 §4、第七章 §1 的核心设计取舍），故取消订单时原料账面不再调整——「待制作 / 制作中」状态原料尚未扣减无须回滚；「待取 / 待配送」状态原料已扣但属于已发生的物理消耗，蛋糕实物处置（第一期只有「报损废弃」一种，第二期补「入成品库存继续售卖」选项）写到 `order_status_log` 备注与第二期的成品库存相关表，不在本表重复记一条原料变动

#### 配方模块
- `recipe`：商品 id 唯一约束（一个商品一份配方）；含制作步骤、损耗率、过敏原
- `recipe_ingredient`：联合唯一约束（配方 id + 原料 id）
- `recipe_snapshot`：订单进入「制作中」状态时锁定配方快照（JSONB），完工扣料按快照口径（**第一期仅现做路径触发快照锁定**；第二期补成品库存后，「从成品库存扣减」路径不进入制作中、不锁定快照，按成品批次本身的配方快照口径扣减，详见需求文档第十三章第 4 节取舍 2）

#### 客户档案模块
- `customer`：wxid 唯一约束
- `customer_phone` / `customer_address` / `customer_note`：列表字段，按 customer_id 关联；customer_note 含来源（手动录入 / 机器人提取）；订单备注优先同步、档案备注异步补偿的提取与提醒机制详见 service 层（数据模型不额外体现「待补写告警」状态，由 service 层异步任务维护）

#### 订单模块
- `order`：含内部 id（自增主键）+ 当日编号（用于显示，格式「月日-流水」）；下单日期 + 当日编号联合唯一索引（编号生成的并发安全）；订单状态字段（枚举：待制作 / 制作中 / 待取 / 待配送 / 配送中 / 已完成 / 已取消）；配方快照 id（制作中状态后填）
- `order_status_log`：每次状态变更留痕，含旧状态 / 新状态 / 触发动作 / 操作人 / 时间 / 备注（取消原因等文本说明）/ `change_diff`（JSONB，记录改单操作前后的字段级 diff，结构化便于回执渲染与事后审计；与「备注」字段分工：备注记取消原因等自由文本，change_diff 记字段级变更）

#### 采购模块
- `purchase`：关联 ingredient；含数量、金额、购买来源、操作人、时间

#### 鉴权模块（v1.0.5 新增）
- `user`：登录用户表，6 位 PIN 码登录体系的数据基础。字段含 user_id（自增主键）/ pin_hash（bcrypt 哈希，不存明文）/ display_name（如「店主」「店员张三」「后厨李四」，写入所有留痕表的「操作人」字段）/ role（枚举：店主 / 店员 / 后厨）/ enabled（账号启用开关）/ failed_attempts（PIN 错误次数，用于锁定判断）/ locked_until（锁定截止时间）/ last_login_at。**注意：bcrypt 哈希同一明文每次结果不同（随机 salt），`pin_hash` 字段上不能加唯一索引、也无法直接 `SELECT` 匹配——登录校验需遍历 `enabled = true` 的用户逐个 `bcrypt.checkpw`；防重复 PIN 不靠数据库约束而靠管理制度（每人独立 PIN，与第六章 §4 安全边界的诚实说明一致）**
- `api_token`：OpenClaw 等外部程序持有的长期凭证表。字段含 token_id（自增主键）/ token_hash（bcrypt 哈希，请求时仅传明文 token、后端哈希后比对）/ name（如「OpenClaw 主机」）/ enabled / created_at / last_used_at。**与 `user` 表分离**：`user` 是给人登录用的（有 pin_hash、failed_attempts、locked_until 等字段对程序无意义），`api_token` 是给程序调用的（无 PIN 锁定逻辑，泄露靠轮换）；凭证类型不同语义不应混淆
- `bot_conversation_log`：机器人对话留痕表，对应需求文档第九章 §4-(5) 五类对话痕迹（店员原始消息 / 识别结果 / 机器人回执 / 店员回复 / 最终入库订单）。字段含 log_id（自增主键）/ session_id（同一次接单流程的多次交互聚合，由 OpenClaw Plugin 从 `ctx.sessionKey` 派生，便于按单追溯）/ step_type（枚举：上述五类）/ payload（JSONB，原始内容）/ operator_wxid（可空，店员侧留痕来源）/ order_id（可空，最终入库时回填关联订单）/ created_at；**写入路径**（v1.0.6 明确）：前四类（店员原始消息 / 识别结果 / 机器人回执 / 店员回复）由 OpenClaw Plugin 注册 `message_received` / `message_sent` / `before_tool_call` hook 后 POST 到 `/api/internal/bot-conversation-log` 内部端点写入（详见第三章 §4）；第五类（最终入库订单）由 MCP 工具 `create_order` 在 service 层事务中直接写入（与订单入库同事务，确保一致性）；用于事后回溯错误环节、优化机器人提示词，**作为系统内部数据保留，不展示给店员**

#### OpenClaw 凭证配置说明
- OpenClaw 持有的 Bearer Token 由 `api_token` 表签发（不挂在 `user` 表上，原因见上）；token 与 PIN 不同源——PIN 用于人登录浏览器，token 用于 OpenClaw 程序调用 MCP；token 泄露时在 `api_token` 表中删除旧记录、签发新 token 并更新 OpenClaw 配置

### 3. 状态机存储

订单状态机的存储形式：`order.status` 枚举字段（待制作 / 制作中 / 待取 / 待配送 / 配送中 / 已完成 / 已取消），配合 `order_status_log` 留痕。

状态变更必须走 service 层事务（更新 status + 写 order_status_log + 触发库存联动 + WebSocket 推送），禁止直接更新 status 字段。每个状态变更动作（开始制作、完工、已交配送、关闭等）对应一个 service 方法，方法是状态变更的唯一入口。

### 4. 关键索引（性能考虑）

- `order(下单日期, 当日编号)` 唯一索引：编号生成的并发安全
- `order(订单状态)` 索引：后厨屏按状态过滤
- `order(客户 id, 下单时间)` 索引：补录时的重复检测
- `ingredient_stock_log(原料 id, 操作时间)` 索引：留痕查询
- `order_status_log(订单 id, 操作时间)` 索引：留痕查询
- `user(display_name)` 索引：按操作人查询留痕（pin_hash 上不加索引——登录是遍历 checkpw，不依赖索引；详见 §4.2 鉴权模块注意事项）
- `api_token(token_hash)` 索引：Bearer Token 校验
- `bot_conversation_log(session_id, created_at)` 索引：按接单会话追溯完整对话流
- `bot_conversation_log(operator_wxid, created_at)` 索引：按店员查历史接单痕迹

## 五、API 契约

### 1. 设计原则

1. **contract-first（前端设计先行）**：先定 API 契约（路径、请求体、响应体），前端可 mock 数据先跑起来，后端按契约实现。具体流程：写 OpenAPI schema → 前端用 `openapi-typescript` 生成 TS 类型 → 前端用 mock server 开发 → 后端按 schema 实现
2. **前后端类型一致**：后端 FastAPI 自动生成 OpenAPI schema，前端用 `openapi-typescript` 自动生成 TS 类型；任何 API 变更先改 schema，前后端同步更新
3. **HTTP API 与 MCP 工具共用 service 层**：不重复业务逻辑；后端管理页面、后厨屏走 HTTP API，机器人走 MCP 工具，三者最终调同一组 service 方法
4. **鉴权双轨（v1.0.5 新增）**：
   - **HTTP API（后端管理页面 + 后厨屏）走 session cookie**：用户在登录页输入 6 位 PIN 码，校验通过后后端签发 session cookie（HttpOnly、Secure、SameSite=Strict），后续 HTTP 请求浏览器自动携带；session 有效期后端管理页面 8 小时、后厨屏 90 天（v1.0.6 从 30 天延长——后厨屏挂在墙上单人使用、不频繁切换用户，延长到 90 天减少重输 PIN 的频次）；差异通过登录时提交 `client_type` 字段区分
   - **MCP 工具（OpenClaw）走 Bearer Token**：OpenClaw 在配置中持有长期 Bearer Token，每次 MCP 调用在 `Authorization: Bearer <token>` header 中携带；token 泄露可在后端轮换
   - **内部回写端点（OpenClaw Plugin）走 Bearer Token**：Plugin 用同样的 Bearer Token 机制回写 `bot_conversation_log`（详见第三章 §4）；与 OpenClaw 调 MCP 共用 token 或单独签发由部署决定，建议同源减少 token 管理面
   - **鉴权由 backend 承担（不在 Caddy）**：Caddy 反代层只做 TLS 终止、端口屏蔽与路径过滤，**不参与业务鉴权**——session cookie 与 Bearer Token 校验全部由 backend 的 FastAPI dependency 统一处理（避免在 Caddy 里维护一套与后端重复的鉴权规则、避免两处配置漂移）；校验失败返回 401（未登录 / session 失效 / token 无效）或 403（已登录但角色无权限）

### 2. HTTP API 清单（后厨屏 + 后端管理页面）

详细路径、请求体、响应体由 OpenAPI schema 定义（开发期自动生成、前端类型同步）。这里只列清单。

**全局鉴权说明**：除 `/health/*`（健康检查）与 `POST /api/auth/login`（登录本身）外，**所有 HTTP 端点必须携带有效 session cookie**；**所有 MCP 工具必须携带有效 Bearer Token**；**所有 `/api/internal/*` 端点必须携带有效 Bearer Token 且不接受 session cookie**（这些端点仅供 OpenClaw Plugin 等程序回写，浏览器路径即便登录也无权访问）。未携带、失效或锁定中返回 401；已登录但角色无权限返回 403。

#### 鉴权模块（`/api/auth/*`，v1.0.5 新增）
- `POST /api/auth/login`：请求体含 `pin`（6 位数字）+ `client_type`（admin / kitchen）；校验通过签发 session cookie 并返回当前用户基本信息；PIN 错累加 `user.failed_attempts`，错 5 次置 `locked_until = now + 10min`，期间登录返回 429
- `POST /api/auth/logout`：清除当前 session
- `GET /api/auth/me`：返回当前登录用户（display_name / role），前端用于初始化界面与操作人显示

#### 内部回写端点（`/api/internal/*`，v1.0.6 新增）
- `POST /api/internal/bot-conversation-log`：仅供 OpenClaw Plugin 回写对话留痕（详见第三章 §4）；请求体含 `session_id` / `step_type` / `payload` / `operator_wxid` / `order_id`（可空）；后端校验 Bearer Token 后写入 `bot_conversation_log` 表；不接受 session cookie 鉴权（即便用户已登录浏览器也无权直接调）

#### 订单模块（`/api/orders/*`）
- 列表查询（按状态、日期过滤）：后厨屏看板 + 后端管理页面列表共用
- 详情查询（按订单编号）：后厨按编号查单（需求文档第十章核心层 (3)）+ 后端管理页面 + 机器人 `query_order` 共用同一 service
- 创建订单（后端管理页面用，机器人走 MCP）
- 改单（PATCH，仅待制作状态允许）
- 取消订单（按状态决定回滚策略）
- 开始制作（仅现做路径，第一期；按需求文档第十三章第 4 节取舍 2，第一期不提供「从成品库存扣减」入口）
- 完工（扣料 + 切状态；第一期简化口径，按需求文档第十三章第 4 节取舍 1，第二期扩展为两阶段事务）
- 标记已交配送（待配送 → 配送中；需求文档第十章 §3 核心层 (4)e 明确「后厨或店员」均可触发，故 HTTP 端后厨屏 / 后端管理页面均可调用，机器人侧另有 `mark_dispatched` MCP 工具）
- 客户到店取货关闭（两个子场景：合并按钮 / 仅交付）
- 配送订单送达关闭（两个子场景：送达付款 / 已付送达；主入口为机器人 `close_delivery_order`，后端管理页面同步提供作为「机器人功能镜像」）

#### 客户档案模块（`/api/customers/*`）
- 按 wxid 查询
- 详情查询
- 新建（首次下单建档）
- 追加电话/地址/备注

#### 菜单模块（`/api/menu/*`，店主后台 CRUD）
- 列表、新增、修改、上下架

#### 原料模块（`/api/ingredients/*`，店主后台 CRUD）
- 档案列表、新增、修改
- 库存调整（盘点 / 报损）
- 采购登记（`/api/purchases`）

#### 配方模块（`/api/recipes/*`，店主后台 CRUD）
- 列表、新增、修改
- 修改影响提示（哪些待制作订单的缺料检测会变化）

#### 后厨屏专属（`/api/kitchen/*`）
- 今日订单看板（按取货时间排序）
- 按编号查单（复用 `/api/orders/{编号}` 端点，需求文档第十章核心层 (3)）；后厨屏提供专门入口，不重复造端点
- WebSocket `/api/kitchen/ws`：新订单推送 + 提示音播报触发

### 3. MCP 工具清单

见第三章 MCP 工具清单表。每个工具对应一个 service 方法。

### 4. 错误码与返回格式

统一响应格式（成功）：
```json
{ "data": "<业务数据>", "error": null }
```

统一响应格式（失败）：
```json
{ "data": null, "error": { "code": "ORDER_NOT_FOUND", "message": "订单 0628-032 不存在", "details": {} } }
```

- 错误码用大写下划线（机器可读），HTTP 状态码遵循 RESTful（200/400/404/500）
- 业务错误（订单状态不允许操作、商品下架、缺料等）返回 400 + 业务错误码
- 鉴权错误：401（未登录 / session 失效 / Bearer Token 无效）+ `UNAUTHORIZED`；403（已登录但角色无权限）+ `FORBIDDEN`；429（PIN 错 5 次触发锁定、API 限流）+ `RATE_LIMITED`
- 系统错误（DB 故障、未知异常）返回 500 + `INTERNAL_ERROR`，详细堆栈写日志不暴露给前端

**客户档案备注异步补偿的失败处理（v1.0.5 明确）**：订单备注随订单同步入库（同事务，失败回滚整单）；档案备注通过 FastAPI BackgroundTasks 异步落账，重试 3 次后仍失败**仅写 structlog WARN 日志**（含订单 id、客户 wxid、待写入的备注内容、失败原因），**不推 webhook**——业主认为小店场景每周查日志即可，webhook 噪音大于价值。系统级关键错误（DB 故障、订单创建失败、库存扣减异常等）仍走第七章 §6 的 webhook 通道推运维群，二者应用场景明确区分。

## 六、操作人留痕方案

### 1. 三入口操作人策略

| 入口 | 操作人来源 | 持久化方式 |
|---|---|---|
| 机器人（店员侧） | 店员微信 wxid | OpenClaw 调用 MCP 工具时自动传入，后端记录到留痕；对话痕迹由 OpenClaw Plugin 通过 `message_received` / `message_sent` / `before_tool_call` hook 回写（详见第三章 §4） |
| 后厨屏（后厨侧） | PIN 码登录（session 90 天，v1.0.6 从 30 天延长） | session cookie，登录用户的 `display_name` 即操作人，写入所有留痕表 |
| 后端管理页面（店主/店员侧） | PIN 码登录（session 8 小时） | session cookie，登录用户的 `display_name` 即操作人，写入所有留痕表 |

### 2. PIN 码登录流程（v1.0.5 替换原后厨屏 X-Kitchen-Operator 自报机制）

后端管理页面与后厨屏共用同一套登录接口（`POST /api/auth/login`，详见第五章 §2），仅 `client_type` 字段不同导致 session 有效期不同：

1. **首次打开**：浏览器加载前端 → 检测无有效 session cookie → 重定向到登录页
2. **输入 PIN**：用户输入 6 位数字 PIN（界面提供数字键盘，方便触屏）；前端 POST `/api/auth/login`，请求体含 `pin` + `client_type`（admin / kitchen）
3. **后端校验**：
   - 遍历 `enabled = true` 的 `user` 记录，对每条 `pin_hash` 用 `bcrypt.checkpw` 校验（bcrypt 同一明文每次哈希结果不同，不能直接 SELECT 匹配，详见 §4.2 鉴权模块注意事项）；命中即登录用户
   - 若 `locked_until > now()`，返回 429 + `RATE_LIMITED`，提示「账号已锁定，请 X 分钟后重试」
   - 若 PIN 错误：`failed_attempts += 1`；达到 5 次置 `locked_until = now() + 10min`、重置 `failed_attempts = 0`；返回 401 + `UNAUTHORIZED`
   - 若 PIN 正确：重置 `failed_attempts = 0`、`locked_until = NULL`、更新 `last_login_at`；签发 session cookie（HttpOnly、Secure、SameSite=Strict），有效期 admin=8h、kitchen=90d（v1.0.6 从 30d 延长——后厨屏挂墙上单人使用、不频繁切换用户，延长减少重输 PIN 频次）
4. **限流双维度 + 单 IP 兜底**：API 层基于「客户端 IP + 用户名（PIN 对应的 display_name）」双维度限流——浏览器路径与脚本直打 API 路径都受同一锁定逻辑约束，防止攻击者绕过浏览器直接爆破 API。**注意双维度的缝隙**：攻击者从同一 IP 对不同用户各试 5 次，单 IP 可发起 `5 × 用户数` 次尝试；小店用户数少（5-10 人）绝对值可控，但仍设**单 IP 每日上限 50 次**兜底，超出即拒绝该 IP 当日所有登录请求
5. **后续操作**：所有 HTTP 请求浏览器自动携带 session cookie；后端 FastAPI dependency 校验 session，从 session 中取出 `display_name` 作为操作人写入留痕表
6. **后厨屏切换操作人**：后厨多人共用同一平板时，提供「切换用户」入口（清当前 session 重新输 PIN），日常靠 90 天长期 session 不频繁重输
7. **v1.0 设计废弃声明**：原 v1.0 §6-2「后厨屏首次配置 + localStorage + `X-Kitchen-Operator` header」机制**全部废弃**。理由：(a) 自报性质无法防恶意，业主挑战「谁拿到 URL 都能进后台太离谱」；(b) PIN 码登录后操作人来源天然落在 session user 上，无需独立配置流程；(c) localStorage 易被本地篡改，session cookie（HttpOnly）安全性更高

### 3. 留痕粒度

- 订单创建、改单（含 `change_diff` 字段级 diff）、取消、状态切换：写入 `order_status_log`
- 原料库存调整、报损、盘点校正：写入 `ingredient_stock_log`
- 采购登记：`purchase` 表自带「操作人」字段
- 机器人对话留痕：`bot_conversation_log`（覆盖五类对话痕迹，前四类由 OpenClaw Plugin 通过 hook 回写、第五类由 MCP 工具 create_order 写入，详见第三章 §4 与第四章 §2）
- 所有留痕表的「操作人」字段**统一从当前 session user 或 MCP 调用上下文中的 wxid 推导**，前端无权自报；后厨屏、后端管理页面操作人即登录的 `display_name`，机器人操作人即店员 wxid

### 4. 安全边界说明（v1.0.5 重写）

**安全机制（四层叠加）**：
1. **6 位 PIN 码**：100 万种组合，bcrypt 哈希存储（不存明文）——**白话**：即使有人偷走数据库，也看不到原始 PIN，只能看到一串乱码
2. **API 层双维度限流 + 单 IP 兜底**：错 5 次锁 10 分钟，IP + 用户名双维度，浏览器路径与 API 直打路径难度等同，单 IP 每日上限 50 次兜底——**白话**：暴力穷举 PIN 会被自动锁住，攻击者无法靠「试到对为止」破解
3. **HTTPS（Caddy 自动 TLS）**：传输加密，防嗅探；公网域名备案完成后由 Caddy 自动从 Let's Encrypt 签发与续签证书（客户端无需任何配置）；备案期间用自签证书兜底，客户端需一次性导入 CA 信任（流程见第七章 §2）——**白话**：店里 WiFi 上有人偷听网络流量，看到的也是加密后的乱码，看不到 PIN 或订单内容
4. **Bearer Token（OpenClaw 侧）**：长期 token 与人登录 PIN 不同源，OpenClaw 主机泄露时仅暴露 token，可在后端轮换——**白话**：OpenClaw 程序调用接口用的是另一把钥匙，跟店员输的 PIN 互不影响

**防爆破分析（6 位 PIN，针对第 1+2 层）**：本节分析针对「PIN 码本身被暴力穷举」的场景（即第 1 层 6 位组合 + 第 2 层限流的相互作用）；第 3 层 HTTPS 防嗅探、第 4 层 Bearer Token 防 OpenClaw 凭证泄露，与 PIN 爆破不直接相关。
- 浏览器路径：错 5 次锁 10 分钟 → 爆破 100 万种组合需 100 万 / 5 × 10 分钟 ≈ **3.8 年**
- API 直打路径：受同一锁定逻辑约束 + 单 IP 每日 50 次兜底 → 等同浏览器路径难度甚至更难
- 结论：在限流到位的前提下，6 位 PIN 在爆破层面是安全的

**固有弱点诚实说明**（这些是短密码共有缺陷，非 PIN 码特有）：
- **肩窥**：4-6 位数字短，店员输入时被旁人看到容易记下。所有短密码都有此问题，无法靠技术完全防住；缓解靠输 PIN 时遮挡、屏幕角度等物理手段
- **便签泄漏**：员工记不住 PIN 可能写在便签贴电脑边——这是管理问题，店主制度上禁止
- **店主懒政共用 PIN**：若店主给所有员工配同一 PIN，留痕退回到粗粒度。要靠制度确保每人独立 PIN
- **公网定向攻击**：架构 A 下系统经公网暴露，但靠第六章 §4 多层安全机制（HTTPS + PIN + Bearer Token + 限流）+ Caddy 仅放行必要路径防御；与认证方式无关的部分靠云服务商的安全组、防火墙配置兜底
- **最终兜底**：上述弱点一旦被利用，操作人留痕是事后追溯的最后防线——真出事能查到「这单改单时登录的是谁的 PIN」

**v1.0 设计废弃声明**：原 v1.0 §6-4「本系统无账号体系、无密码验证，操作人字段是自报性质，物理访问控制已经足够」的论述**全部废弃**。该论述低估了「客户档案 + 订单金额 + 客户电话」等敏感数据的风险——架构 A 全云部署下系统经公网暴露，任何能访问 URL 的人（不只是店内人员）都能查看与修改敏感数据；即便后续切回店内本地服务器，客户连店内 WiFi、设备被借用、路由器被蹭网等场景下，「无认证 + 自报」同样意味着物理访问边界一旦被突破即可全量访问。

**WebAuthn（指纹/面部）作为第二阶段可选叠加项**：浏览器原生支持 WebAuthn 标准，注册一次指纹后免 PIN 登录，体验比输 PIN 更好、防肩窥能力更强（生物特征偷看无用）。但当前不强制：(a) 每台设备需分别注册一次指纹；(b) 后厨多人共用平板时要在平板上录多个指纹；(c) 老员工指纹磨损可能识别困难；(d) 旧浏览器可能不支持。第一期上线 PIN 码即可，未来员工反馈输 PIN 麻烦时再加 WebAuthn，**数据模型与 API 设计已预留扩展空间**（`user` 表可加 `webauthn_credential` 字段，登录接口可加 WebAuthn assertion 校验分支）。

## 七、部署与 CI/CD

### 1. Docker 镜像构建

- `Dockerfile.backend`：基于 `python:3.11-slim`，多阶段构建（依赖安装 + 应用打包 + 前端构建产物复制）
- 前端构建产物（`frontend/dist/`）打进 backend 镜像，由 FastAPI 直接 serve（单镜像部署，简化）
- `Dockerfile.caddy`（v1.0.5 新增，v1.0.6 调整）：基于 `caddy:2` 官方镜像，**不再 baked 自签证书文件**（v1.0.6 改为 Let's Encrypt 自动签发，Caddy 容器内无需预置证书），仅含 Caddyfile 与持久化目录挂载；备案期间如需自签证书兜底，由 Caddy 的 `tls internal` 指令在容器内自动生成内部 CA 与证书，无需手工 mkcert
- 备选：前后端分离部署，前端单独 Nginx 镜像

### 2. docker-compose 编排

`deploy/docker-compose.yml` 大致结构（v1.0.6 按架构 A 调整 caddy 服务）：

- `caddy` 服务：基于 `caddy:2` 官方镜像，**对外暴露 443 与 80 端口**（80 仅做 301 跳转到 443 + ACME HTTP-01 挑战用），反向代理到 `backend:8000`；Caddyfile 配置域名（备案完成后切换为正式域名）+ `tls` 指令（默认自动 Let's Encrypt；备案期用 `tls internal` 切到自签）+ 反代规则 + 超时 + 日志；挂载 `caddy_data` 卷持久化 ACME 账户与证书状态（避免每次重启重新签发）
- `backend` 服务：构建自 `Dockerfile.backend`，**8000 端口仅容器网络内可达，不直接对外暴露**（`expose: [8000]` 而非 `ports`）；依赖 postgres；环境变量含 `DATABASE_URL`、`SESSION_SECRET`（签发 session cookie 用）、`OPENCLAW_BEARER_TOKEN`、`ALLOWED_HOSTS`（Caddy 反代后 backend 只接受来自 Caddy 的请求，防绕过反代直连）
- `postgres` 服务：`postgres:15` 镜像，**仅容器网络内可达**，数据卷持久化（`pgdata`），环境变量含密码
- `volumes`：`pgdata` 持久化数据库；`caddy_data` 持久化 Caddy ACME 账户与证书状态；`caddy_config` 持久化 Caddy 配置
- **OpenClaw 不进 docker-compose**：OpenClaw 是独立进程，由业主已有的部署方式（systemd / supervisor / 手动等）管理；OpenClaw 配置中 MCP server URL 指向 `https://cake-shop.example.com/mcp/sse`、Bearer Token 写在配置中

**TLS 证书策略（v1.0.6 重写，双轨自动）**：
- **域名备案完成后（正式运行）**：Caddyfile 配置正式域名 + 默认 `tls`（无参数），Caddy 自动向 Let's Encrypt 申请证书、自动续签、自动 HTTPS 重定向；客户端（店主电脑、店员手机、后厨平板、OpenClaw 主机）无需任何 CA 导入操作
- **备案期间（1-3 周过渡）**：先用云服务商提供的公网 IP + Caddyfile 配置 `tls internal`，Caddy 用内部 CA（Caddy 自带）签发自签证书；客户端首次访问会收到浏览器证书警告，需要一次性手动信任（成本因设备类型显著不同，业主需提前知晓）：
  - **桌面端**（店主电脑，Windows/macOS）：浏览器点「高级 → 继续访问」即可，约 30 秒；或导入 Caddy 内部 CA 到系统信任库（约 2-3 分钟）一劳永逸
  - **iOS / Android**：浏览器层面「继续访问」难度因浏览器而异（Safari/Chrome 政策不同），国产 ROM 上部分浏览器可能直接拒绝自签证书无法绕过；建议备案期间店内主要用桌面端，移动端等域名备案完成后启用
  - **后厨平板**：选店内固定一台平板，导入 CA 信任（一次性），之后无感
  - 备案完成、Caddyfile 切回正式域名后，所有客户端自动走 Let's Encrypt 证书、无需任何后续操作
- **Caddy 自动续签**：Let's Encrypt 证书有效期 90 天、Caddy 在到期前 30 天自动续签；不依赖 certbot 等外部工具、无需 cron 维护

**OpenClaw Bearer Token 配置**：在 OpenClaw 配置文件中注册本系统 MCP server：
```
openclaw mcp add --transport sse https://cake-shop.example.com/mcp/sse \
  --header "Authorization: Bearer <OPENCLAW_BEARER_TOKEN>"
```
token 由店主在后端 `api_token` 配置中生成、写在 OpenClaw 配置中；泄露时在后端轮换并更新 OpenClaw 配置。

**OpenClaw Plugin 部署**：本项目维护的对话留痕 Plugin（详见第三章 §4）随 OpenClaw 配置一起部署——OpenClaw 配置中 `plugins.entries.<plugin-id>` 启用，Plugin 持同样的 Bearer Token（或单独签发）回写到 `/api/internal/bot-conversation-log`。

生产环境用 `docker-compose.prod.yml` override：禁用热重载、限制资源、启用日志收集。

### 3. CI/CD 流水线（GitHub Actions）

`deploy/.github/workflows/ci.yml`，触发条件 `push to main` 或 `tag v*`：

**push to main（持续集成）**：
- 后端：`ruff` lint → `pytest` 测试 → 构建 Docker 镜像 → push 到 GitHub Container Registry

- **开发**：本地 `docker-compose up`，热重载（FastAPI `--reload`、Vite HMR），数据库用独立容器
- **测试**：CI 中 Docker 容器跑测试，每次 PR 触发
- **生产**：云上单台服务器，`docker-compose -f docker-compose.prod.yml up -d`，无热重载；OpenClaw 与 docker-compose 同机部署但作为独立进程

- PostgreSQL 数据每日备份：`pg_dump` + 文件保留 7 天
- 备份脚本由 cron 触发，备份文件可配置上传到店主云盘（百度网盘 / 阿里云盘等）
- 恢复演练：每月一次，从备份还原到测试环境，验证可用

### 6. 监控与告警（轻量）

小店场景不引入 Prometheus / Grafana 等重量级监控，采用：
- 后端日志：`structlog` 输出 JSON 日志，docker-compose 集中收集到 `logs/` 目录；客户档案备注异步补偿失败、PIN 错误等次要事件仅写日志（不推 webhook）
- **系统级关键错误**：service 层捕获 DB 故障、订单创建失败、库存扣减异常、Alembic 迁移失败等系统级问题时，通过 webhook 推送到运维微信群（与需求文档第十一章运维群通道一致），让店主第一时间感知系统不可用并启动退阶
- **健康检查分层**（v1.0.5 明确）：
  - `/health/live`：进程存活检查（FastAPI 进程能响应即可），用于 docker-compose healthcheck + 自动重启
  - `/health/ready`：依赖检查（含 DB ping），用于判断 backend 是否真正能服务请求；Caddy 反代层可据此做故障转移（虽然单 backend 节点暂无转移目标，但 ready 端点本身可用于运维排查）
  - 两个端点都不要求鉴权（在 §5.2 全局鉴权说明中显式排除）

## 八、技术细节决策（v1.0.5 集中说明，v1.0.6 增补两项）

本章节集中记录实现期的 13 项纯技术决策，避免散落在各章节难以追溯。这些决策不涉及业务取舍（业务取舍由需求文档承担），是技术专家基于「小店场景、单人开发、架构 A 全云部署」前提下的最优雅选择。

**决策性质说明**：这 13 项中，**§8.4 / §8.10 / §8.11 三项是 v1.0.5 新拍**（与 v1.0.5 核心修订直接相关，如客户备注异步补偿失败仅写日志、机器人对话留痕落表、改单前后对比存储）；**§8.9 / §8.11 在 v1.0.6 又做了重要调整**（语音播报改为提示音 mp3、bot_conversation_log 改为 Plugin hook 写入）；**§8.12 / §8.13 是 v1.0.6 新增**（原料冷启动策略、后台播报设置面板）；**其余 8 项是 v1.0~v1.0.4 隐含但未集中说明的实现期决策**（如缺料检测事务、订单编号并发、WebSocket 单 worker、Alembic 迁移纪律等），集中收录是为后续开发提供单一可信源追溯，不代表这些都是 v1.0.5/v1.0.6 新增的设计。

### 1. 缺料检测的事务隔离

订单创建 + 承诺量（「待制作 + 制作中」订单对该原料的需求总和）读取放**同一数据库事务**中，对相关原料行执行 `SELECT … FOR UPDATE` 加行锁，确保两个并发订单不会同时读到未包含对方的承诺量（需求文档第十一章 §4-已知风险 3 的竞态风险）。

**不上 SERIALIZABLE 隔离级别**：性能开销大、PostgreSQL 在 SERIALIZABLE 下冲突重试不可控；FOR UPDATE 行锁更精确、对小店流量足够。

**不上消息队列串行化订单创建**：单店物理上不存在真正高并发（一次只有一个店员接单），队列是过度设计。

### 2. 订单编号的并发安全

订单编号「月日-流水」生成走「事务内 `SELECT MAX(当日流水) FOR UPDATE` + 唯一索引兜底 + 应用层重试」三道保险：
1. 事务内对当日已有订单的流水号加锁后取 MAX+1，保证并发事务串行生成编号
2. `(下单日期, 当日编号)` 唯一索引兜底，万一锁失效撞键抛 IntegrityError
3. service 层捕获 IntegrityError 后回滚 + 重试（最多 3 次），仍失败返回 500

**不用 PostgreSQL 序列**：序列跨日重置需额外维护，且与「月日-流水」的语义不对齐；MAX+1 + 锁更直观。

### 3. WebSocket 多 worker 部署

backend 容器 Uvicorn **单 worker** 部署（`--workers 1`）。WebSocket 连接是进程本地的，多 worker 下后厨屏连在 worker A、service 在 worker B 触发广播会收不到；单 worker 部署天然消除这一问题。

小店流量极低（每天几十单），单 worker 性能完全够用。**将来真的需要横向扩展时再引入 Redis Pub/Sub 做跨 worker 广播**，当前是 YAGNI。

### 4. 改单前后对比的存储

`order_status_log.change_diff` 用 **JSONB 字段**存储改单前后的字段级 diff（如 `{"quantity": {"old": 1, "new": 2}, "pickup_time": {"old": "15:00", "new": "16:00"}}`）。JSONB 便于回执渲染、事后审计、按字段查询。

**与「备注」字段分工**：备注记取消原因、改单说明等自由文本；`change_diff` 记结构化字段变更。两者并存，互不替代。

### 5. 配方修改影响提示的查询口径

「修改影响提示」查询范围 = **所有「待制作 + 制作中」状态的订单**，按「商品 × 数量 × 配方原料」维度计算影响。

**与需求文档第五章 §5-2 承诺量口径一致**：承诺量只覆盖现做路径（待制作 + 制作中）的原料占用，「待取 / 待配送 / 配送中」订单的原料消耗已在成品诞生时扣过、反映在当前库存中。配方修改的影响范围与承诺量口径对齐，避免出现「承诺量算了一套、配方影响又算了另一套」的双轨混乱。

### 6. Alembic 迁移纪律

所有生产数据库迁移必须**前向兼容**（forward-compatible）：
- 加字段：必须可空或有默认值，旧代码不读该字段也能运行
- 删字段：**三步走**——(a) 标记 deprecated（代码不再写但仍读）→ (b) 双写过渡（新旧字段并存，定时同步）→ (c) 物理删除（确认无依赖后）
- 改字段类型：先加新字段、双写、迁移数据、再删旧字段

**理由**：第七章 §3 的「tag v* 失败回滚保留上一版本镜像」策略下，镜像回滚后旧代码可能跑在新数据库上。前向兼容保证旧代码在新库上不崩。**回滚只回镜像，不回数据库结构**。

### 7. 平板保障（息屏 / 唤醒 / 断网重连）

后厨屏「订单不漏」是系统核心目标。架构 A 全云部署下后厨屏通过公网 WSS（经 Caddy）与 backend 通信，网络可靠性比店内本地部署更关键，需要以下浏览器层保障：
- **Wake Lock API**（`navigator.wakeLock.request('screen')`）：保屏幕常亮，防止平板系统息屏后 WebSocket 断开；失活时（如切换 tab）自动重新申请
- **WebSocket 指数退避自动重连**：连接断开后 1s → 2s → 4s → 8s → 最大 30s 重试（架构 A 下公网链路抖动比局域网频繁，指数退避避免短时间狂重试）；重连成功后立即主动拉取一次最新订单列表（避免漏掉断连期间产生的新单）
- **Page Visibility API**（`document.visibilityState`）：平板被切到后台或锁屏后回到前台时，主动拉取一次最新订单（与 WebSocket 重连形成双保险）
- **断网降级提示**：WebSocket 连续重试超过 5 次未成功时，后厨屏界面顶部显示「网络异常，正在重连…」红色横幅提示后厨人工留意微信群接单消息（对接需求文档第十一章场景 1 退阶路径）；网络恢复后横幅自动消失
- **音频自动播放策略**：首次用户交互（任意触摸，包括登录时输 PIN）后才能触发 `<audio>` 元素播放，前端在登录后引导一次触摸以满足浏览器策略（详见 §8.9）

### 8. 健康检查分层

- `/health/live`：进程存活检查（FastAPI 进程能响应即可），用于 docker-compose healthcheck + 自动重启
- `/health/ready`：依赖检查（含 DB ping），用于判断 backend 是否真正能服务请求

两个端点都不要求鉴权。**不引入 Prometheus / Grafana 等重量级监控**（见 §7.6 监控策略）。

### 9. 后厨屏新单提示（v1.0.6 重写：提示音 mp3 + 屏幕高亮）

后厨屏收到新订单时，触发两层提示：
- **提示音 mp3**：前端预置一个 1-2 秒的提示音文件（如经典的「叮」声，从免费音效库如 freesound.org 下载或自录），通过 `<audio>` 元素播放；不读商品名、不读数量、不读取货时间——业主反馈「就一句提示就可以了，甚至不读文字都可以」，原 v1.0.5 的 Web Speech API TTS 方案在嘈杂后厨场景实际听不清、且不同浏览器/操作系统的中文 TTS 音色与质量差异大，过度设计无收益
- **屏幕高亮**：新订单在订单看板顶部以高对比色（如黄色边框 + 轻微脉冲动画）持续高亮 30 秒，确保即使错过提示音、后厨抬头看屏也能立刻注意到；高亮在用户点击查看详情后消失

**为什么不用 Web Speech API TTS**：(a) 后厨环境嘈杂，合成的中文语音听不清，提示音 + 屏幕高亮反而更有效；(b) 跨浏览器/操作系统的中文 TTS 质量参差不齐，macOS / Windows / Android / iOS 音色差异大，难以保证一致体验；(c) 业主明确反馈不需要读文字。简化后实现成本与维护成本都更低。

**音频自动播放限制**：浏览器禁止无用户交互的自动播放音频，前端在登录时引导后厨「点一下屏幕开始接单」触发首次用户交互，后续 `<audio>` 播放即可被允许（详见 §8.7 平板保障）。

### 10. 客户档案备注异步补偿

订单备注随订单同步入库（同事务，失败回滚整单）；档案备注通过 **FastAPI BackgroundTasks** 异步落账：
1. 订单入库事务提交后，注册 BackgroundTask 写档案备注
2. 任务内重试 3 次（间隔 1s / 2s / 4s）
3. 仍失败**仅写 structlog WARN 日志**（含订单 id、客户 wxid、待写入的备注内容、失败原因），不推 webhook

**不引入 Celery / Redis 队列**：小店场景 BackgroundTasks 进程内异步足够，独立 worker 是过度设计。失败靠店主每周或异常时查日志发现，与 §7.6 系统级关键错误的 webhook 通道明确区分。

### 11. 机器人对话留痕的存储位置（v1.0.6 改为 Plugin hook 写入）

机器人对话留痕落在**后端 `bot_conversation_log` 表**（结构见第四章 §2），覆盖需求文档第九章 §4-(5) 五类对话痕迹（店员原始消息 / 识别结果 / 机器人回执 / 店员回复 / 最终入库订单）。

**写入路径（v1.0.6 重新设计）**：前四类由 **OpenClaw Plugin** 注册 `message_received` / `message_sent` / `before_tool_call` hook 后通过 `/api/internal/bot-conversation-log` 内部端点写入（详见第三章 §4）；第五类「最终入库订单」由 MCP 工具 `create_order` 在 service 层事务中直接写入（与订单入库同事务，确保一致性）。Plugin hook 的确定性优势与失败降级论述见第三章 §4，此处不重复。

**不依赖 OpenClaw 侧日志**：OpenClaw 自带的 LLM 调用日志是调试性质（记录 prompt / completion 原文），用途与业务留痕不同；两者并存不冲突，但**业务留痕以后端为单一可信源**——后续无论换 LLM、换 agent 框架、换 hook 实现，业务留痕口径不变。

**留痕数据作为系统内部数据保留，不展示给店员**（需求文档明确要求）；店主可在数据库层直接查询（事后追溯错误环节、优化机器人提示词）。

**v1.0.5 原方案废弃声明**：原 v1.0.5 设计「让 OpenClaw 在 LLM prompt 中被要求主动调用 `log_conversation_step` MCP 工具」**全部废弃**。理由：(a) 依赖 LLM 自觉不可靠；(b) 每条对话多一次 MCP 调用增加延迟；(c) Plugin hook 方案在确定性、性能、可维护性上都更优。

### 12. 原料冷启动策略（v1.0.6 新增）

系统首次启用时，原料档案与初始库存的录入策略：

1. **原料档案手工录入**：店主在后端管理页面「原料模块」逐条录入原料档案（原料名、单位、安全库存等），不提供批量导入工具——小店原料种类有限（通常 30-50 种），手工录入 1-2 小时内可完成，批量导入（CSV/Excel）需要额外的字段映射、错误处理、回滚机制，过度设计
2. **初始库存通过一次性盘点登记**：原料档案录入完成后，店主实地盘点仓库、在后端管理页面「原料库存调整」一次性录入实际数量；该次调整写入 `ingredient_stock_log` 表、来源类型为「盘点校正」（与日常运营中的盘点入口共用同一 service 方法、同一留痕字段，无需单独的「初始化」入口）。**注意**：盘点仅录入原料的绝对数量、不依赖配方或触发缺料检测，故可在配方档案录入前完成；缺料检测在第一笔订单接单时才被调用，那时配方与初始库存都已就绪
3. **配方档案同步录入**：原料档案就绪后，店主在「配方模块」按商品录入配方（原料 × 用量 × 损耗率）；配方是缺料检测与完工扣料的依据，必须在第一笔订单接单前完成

**不提供「从采购历史反推初始库存」**：(a) 第一期采购登记从系统启用开始才有记录，无法回溯启用前的存量；(b) 反推算法需要假设「启用至今未消耗」，与实际业务不符；(c) 实地盘点一次更准确。

**冷启动期提示**：前端在系统首次登录时检测 `ingredient` 表与 `recipe` 表是否为空，若为空则在首页显著位置提示「请先完成原料档案与配方录入，再开始接单」，引导店主按顺序完成冷启动；提示在录入第一份配方后自动消失。

### 13. 后厨屏播报设置面板（v1.0.6 新增）

后厨屏提供「设置」入口（PIN 码登录后可见，店主角色才能修改、后厨角色只读），允许配置：
- **提示音开关**：嘈杂时段（如周末高峰）可临时关闭提示音、仅靠屏幕高亮；非高峰时段重新打开
- **提示音音量**：0-100 滑块，记忆在 localStorage（后厨屏单人使用、无需按用户区分）
- **屏幕高亮持续时间**：默认 30 秒，可调到 10-120 秒；过短后厨可能错过、过长干扰后续订单视觉
- **断网降级提示开关**：默认开启；若后厨网络长期不稳定、横幅频繁出现干扰，可关闭（但 不建议——关闭后断网期间后厨无任何感知）

**为什么放后厨屏而不放后端管理页面**：(a) 这些是后厨屏运行时行为、与后端业务无关；(b) 后厨在使用过程中要根据现场情况（嘈杂程度、人手）即时调整，放在后厨屏减少切换；(c) 后端管理页面是店主/店员侧的订单管理入口，混入后厨屏设置会职责不清。

**设置项持久化**：后厨屏运行在固定平板上、单人使用，设置项记忆在浏览器 localStorage 即可（无需落到后端数据库）；更换平板时重新设置一次成本可接受（约 30 秒），无需同步机制。
