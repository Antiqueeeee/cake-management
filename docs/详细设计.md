## 0. 文档说明

### 0.1 本文档的位置

| 文档 | 回答的问题 | 读者 |
|---|---|---|
| 需求文档 | 业务要什么、验收标准是什么 | 业主、产品、开发 |
| 技术方案 | 为什么这么选（FastAPI 不选 Django…）、整体架构、模块怎么分 | 开发、架构评审 |
| **详细设计（本文档）** | **具体怎么落地：每个表每个字段、每个状态怎么转、每个页面长啥样** | **即将开工的开发者** |

简单说：详细设计是「写代码的依据」。开发者拿到这份文档 + 技术方案 + 需求文档，应该不用再问任何业务/技术决策细节就能动手。

### 0.2 v1.0 范围

v1.0 只写**数据库详细 schema**（§1）。理由：ORM 模型、Alembic 迁移、service 层业务逻辑都建立在字段清单之上；字段定错了，所有上层建筑都要返工。这是开工前最关键的依赖。

后续版本计划：
- v1.1：§2 状态机（订单状态转换图、触发角色、联动动作）
- v1.2：§4 前端页面与 ASCII 线框（API 详细契约不写——Q1 已确认让 Pydantic 自动生成 OpenAPI、前端 `openapi-typescript` 反生成类型）
- v1.3：§5 第一刀切片任务分解 + §6 开发顺序建议

### 0.3 与技术方案、需求文档的关系

- **技术方案 §4**：给出表清单、关键约束、关键索引（约束级）
- **需求文档各章 §2/§3**：给出业务字段定义（业务级）
- **本文档 §1**：把两者结合，给出可落地的字段清单（落地级）

不冲突：技术方案不重新写字段清单，本文档不重新论证选型与约束。

**冲突时如何裁决**：
- 本文档 ↔ 技术方案冲突：以技术方案为准（约束层高于实现层）
- 本文档 ↔ 需求文档冲突：业务含义以需求文档为准；命名/类型选择以本文档为准（落地选择是本文档职责）

### 0.4 设计约定

**命名**
- 表名 / 字段名：snake_case
- 主键：统一用 `<表名单数>_id`（如 `order_id`、`customer_id`），自增 BIGINT
- 外键：`<引用表单数>_id`
- 时间字段：`*_at` 后缀（`created_at`、`updated_at`、`last_login_at`），类型 TIMESTAMPTZ
- 布尔字段：`is_*` 或 `has_*` 前缀
- 枚举字段：PostgreSQL ENUM 类型，值大写下划线（如 `ORDER_STATUS.PENDING`）

**类型缩写**
- `BIGINT`：64 位整数（主键、外键）
- `INT`：32 位整数（小范围计数）
- `TEXT`：变长字符串
- `NUMERIC(p, s)`：定点数，p 总位数、s 小数位
- `BOOLEAN`：布尔
- `TIMESTAMPTZ`：带时区时间戳
- `DATE`：日期（无时间）
- `JSONB`：二进制 JSON
- `ENUM`：PostgreSQL 枚举类型
- `TEXT[]`：文本数组

**通用字段**（每张表都有，下文表内不再重复说明）
- `<表名单数>_id`：主键，BIGINT NOT NULL，自增
- `created_at`：创建时间，TIMESTAMPTZ NOT NULL DEFAULT now()，不可变
- `updated_at`：更新时间，TIMESTAMPTZ NOT NULL DEFAULT now()，由触发器或 ORM 自动维护

**外键策略**
- 默认 `ON DELETE RESTRICT`（防止误删导致历史留痕悬空）
- 留痕表（`order_status_log`、`ingredient_stock_log`、`bot_conversation_log`）的外键也用 `RESTRICT`——即便关联业务记录被删，留痕必须保留可追溯
- 例外：子表（如 `customer_phone` → `customer`）用 `ON DELETE CASCADE`（父删则子删，避免孤儿数据）

#### 1.2.2 原料模块

##### `ingredient`（原料档案 + 当前库存）

需求文档第五章 §2（档案字段）与 §3（库存字段）合并为一张表——库存当前值与档案 1:1，独立成表徒增 join。

| 列名 | 类型 | 可空 | 默认 | 约束 / 说明 |
|---|---|---|---|---|
| ingredient_id | BIGINT | NO | - | 主键 |
| name | TEXT | NO | - | 原料名，UNIQUE；需求文档第五章 §2-(1) |
| unit | TEXT | NO | - | 计量单位（克/毫升/个/袋）；采购与配方共用 |
| unit_price | NUMERIC(10, 2) | YES | NULL | 最近采购平均单价；无采购记录时为空 |
| safety_stock | NUMERIC(10, 2) | NO | 0 | 安全库存线；低于此值提醒补货，0 = 不提醒 |
| shelf_life_days | INT | YES | NULL | 原料保质期（天）；**仅作记录**，原料不按批次管理、不计算临期（需求文档第五章 §3 末段） |
| storage_location | TEXT | YES | NULL | 存放位置（货架号、冰柜号） |
| common_source | TEXT | YES | NULL | 常用购买来源 |
| current_stock | NUMERIC(10, 2) | NO | 0 | 当前库存；**允许负数**（负数绝对值 = 欠缺量） |
| last_restocked_at | TIMESTAMPTZ | YES | NULL | 最近入库时间 |
| created_at | TIMESTAMPTZ | NO | now() | |
| updated_at | TIMESTAMPTZ | NO | now() | |

**索引**：UNIQUE(name)；BTREE(safety_stock)（低库存预警查询用：WHERE current_stock < safety_stock）

##### `ingredient_stock_log`（库存变动留痕）

| 列名 | 类型 | 可空 | 默认 | 约束 / 说明 |
|---|---|---|---|---|
| log_id | BIGINT | NO | - | 主键 |
| ingredient_id | BIGINT | NO | - | FK → ingredient；ON DELETE RESTRICT |
| change_amount | NUMERIC(10, 2) | NO | - | 本次变动量（正入库 / 负扣减）；CHECK ≠ 0 |
| new_balance | NUMERIC(10, 2) | NO | - | 变动后余额；与 ingredient.current_stock 同步 |
| source_type | STOCK_SOURCE | NO | - | ENUM：PURCHASE / FINISHED_GOODS_DEDUCT / LOSS / INVENTORY_ADJUSTMENT |
| reason | TEXT | NO | - | 必填原因（需求文档第五章 §4 要求） |
| order_id | BIGINT | YES | NULL | FK → `order`；source_type = FINISHED_GOODS_DEDUCT 时关联订单 |
| purchase_id | BIGINT | YES | NULL | FK → purchase；source_type = PURCHASE 时关联采购 |
| operator_user_id | BIGINT | NO | - | FK → user；操作人留痕 |
| created_at | TIMESTAMPTZ | NO | now() | |

**索引**：BTREE(ingredient_id, created_at)；BTREE(source_type)

**约束**：
- CHECK (change_amount <> 0)
- CHECK (
    (source_type = 'FINISHED_GOODS_DEDUCT' AND order_id IS NOT NULL) OR
    (source_type = 'PURCHASE' AND purchase_id IS NOT NULL) OR
    source_type IN ('LOSS', 'INVENTORY_ADJUSTMENT')
  )

**注意**：按技术方案 §4.2，**订单取消不写本表**——原料消耗在完工时已扣且不可回滚，取消订单时原料账面不再调整。

- BTREE(status)：后厨屏按状态过滤
- BTREE(customer_id, created_at)：补录重复检测
- BTREE(pickup_at, status)：后厨今日看板按取货时间排序

**编号生成**（display_no）：**不存字段**，由 SQL 表达式或 ORM property 派生——`to_char(order_date, 'MMDD') || '-' || lpad(daily_seq::TEXT, 3, '0')`，例如 `0629-001`。单一可信源（display_no 是 order_date + daily_seq 的函数，存了易不一致）。

**约束**：
- CHECK (quantity > 0)
- CHECK (
    (pickup_method = 'DELIVERY' AND delivery_address IS NOT NULL) OR
    pickup_method = 'PICKUP'
  )
- CHECK (
    (payment_status = 'PAID' AND payment_method IS NOT NULL) OR
    payment_status = 'UNPAID'
  )

##### `order_status_log`（订单状态变更留痕）

| 列名 | 类型 | 可空 | 默认 | 约束 / 说明 |
|---|---|---|---|---|
| log_id | BIGINT | NO | - | 主键 |
| order_id | BIGINT | NO | - | FK → `order`；ON DELETE RESTRICT |
| old_status | ORDER_STATUS | YES | NULL | 旧状态；首次创建订单时为 NULL |
| new_status | ORDER_STATUS | NO | - | 新状态 |
| action | ORDER_ACTION | NO | - | ENUM：CREATE / UPDATE / START_COOKING / FINISH / MARK_DISPATCHED / CLOSE_PICKUP / CLOSE_DELIVERY / CANCEL |
| change_diff | JSONB | YES | NULL | 改单前后字段级 diff；仅 action = UPDATE 时填，结构见 §1.3.1 |
| note | TEXT | YES | NULL | 自由文本说明（取消原因等） |
| operator_user_id | BIGINT | YES | NULL | FK → user；浏览器端操作时填 |
| operator_wxid | TEXT | YES | NULL | 机器人端操作时填；与 operator_user_id 二选一 |
| created_at | TIMESTAMPTZ | NO | now() | |

**索引**：BTREE(order_id, created_at)

**约束**：
- CHECK (
    (operator_user_id IS NOT NULL AND operator_wxid IS NULL) OR
    (operator_user_id IS NULL AND operator_wxid IS NOT NULL)
  )
- CHECK (
    (action = 'CREATE' AND old_status IS NULL) OR
    (action <> 'CREATE' AND old_status IS NOT NULL)
  )——CREATE 是状态机入口（无 → PENDING）、无旧状态；其他 7 种 action 必须有旧状态

**注意**：
- 所有状态变更必须经 service 层事务（更新 `order.status` + 写本表 + 触发库存联动 + WebSocket 推送），禁止直接改 `status`（技术方案 §4.3）
- `change_diff` 与 `note` 分工：前者记结构化字段变更，后者记自由文本
- **`action = CREATE` 行与 `order.created_at` 互补**：`created_at` 只记时间、无操作人语义；CREATE 行额外承载「谁创建」（含机器人下单时的 `operator_wxid`），是机器人留痕的关键一环，不冗余

### 1.3 JSONB 字段结构

#### 1.3.1 `order_status_log.change_diff`

改单（action = UPDATE）前后的字段级 diff，**仅记实际变更字段**、未变字段不记。下列示例覆盖需求文档第二章故事 5 §311 列出的**全部 7 类可改字段**：商品(`menu_id`) / 数量(`quantity`) / 取货方式(`pickup_method`) / 取货时间(`pickup_at`) / 联系电话(`contact_phone`) / 配送地址(`delivery_address`) / 备注(`order_note`)。`menu_id` 的 diff 内嵌 `name` 字段便于人读、防事后追溯时菜单已改名；实际改单通常只动 1-3 个字段，本例为示意合并展示：

```json
{
  "menu_id": {"old": {"id": 5, "name": "蓝莓蛋糕"}, "new": {"id": 7, "name": "芒果蛋糕"}},
  "quantity": {"old": 1, "new": 2},
  "pickup_method": {"old": "PICKUP", "new": "DELIVERY"},
  "pickup_at": {
    "old": "2026-06-29T15:00:00+08:00",
    "new": "2026-06-29T16:00:00+08:00"
  },
  "contact_phone": {"old": "138xxxx", "new": "139xxxx"},
  "delivery_address": {"old": null, "new": "中关村大街 1 号"},
  "order_note": {"old": null, "new": "加巧克力粉"}
}
```

**字段类型说明**：
- `menu_id` 用 `{id, name}` 嵌套（既可查询又可人读）
- `pickup_method` 用枚举字符串（`PICKUP` / `DELIVERY`）
- `delivery_address` 在自取改配送时通常 `old = null`、配送改自取时 `new = null`

**字段级 diff 的好处**：回执可以渲染「数量从 1 改为 2、取货时间延后 1 小时」；事后审计可精确追溯哪次改了哪个字段。

**与 `note` 字段分工**：`note` 记取消原因等自由文本，`change_diff` 记结构化字段变更，互不重叠。

#### 1.3.2 `recipe_snapshot.content`

订单进入「制作中」时锁定的完整配方内容：

```json
{
  "menu_name": "巧克力蛋糕",
  "steps": [
    {"step_no": 1, "description": "打发蛋白至硬性发泡", "duration_minutes": 10},
    {"step_no": 2, "description": "烘焙 180°C 30 分钟", "duration_minutes": 30}
  ],
  "total_time_minutes": 40,
  "loss_rate": 0.05,
  "allergens": ["乳制品", "麸质"],
  "ingredients": [
    {"ingredient_id": 12, "name": "低筋面粉", "quantity": 200, "unit": "克"},
    {"ingredient_id": 18, "name": "黑巧克力", "quantity": 150, "unit": "克"}
  ]
}
```

**为什么快照**：配方随时可改，但订单按制作那一刻的配方扣料、按当时的损耗率算成本；快照确保扣料口径不被后续配方修改污染。

#### 1.3.3 `bot_conversation_log.payload`

按 step_type 不同结构不同。JSONB 不强约束子结构（不同 step_type 不同结构），由应用层保证写入格式正确：

```json
// step_type = CUSTOMER_ORIGINAL_MSG（店员原始消息）
{"text": "小张订1个巧克力蛋糕，明天下午3点取，电话138xxxx"}

// step_type = RECOGNITION_RESULT（识别结果）
{
  "customer_wxid": "wxid_abc123",
  "menu_name": "巧克力蛋糕",
  "quantity": 1,
  "pickup_at": "2026-06-30T15:00:00+08:00",
  "contact_phone": "138xxxx",
  "missing_fields": []
}

// step_type = BOT_REPLY（机器人回执）
{"text": "已下单，订单号 0629-001"}

// step_type = CUSTOMER_REPLY（店员回复）
{"text": "是的，按这个"}

// step_type = ORDER_PERSISTED（最终入库订单）
{"order_id": 42, "order_no": "0629-001"}
```

#### 1.3.4 `recipe.steps`

配方制作步骤数组（同时作为 `recipe.steps` 字段结构与 `recipe_snapshot.content.steps` 元素结构）：

```json
[
  {"step_no": 1, "description": "打发蛋白至硬性发泡", "duration_minutes": 10},
  {"step_no": 2, "description": "烘焙 180°C 30 分钟", "duration_minutes": 30},
  {"step_no": 3, "description": "出炉冷却", "duration_minutes": 15}
]
```

每步含序号、描述、可选耗时（需求文档第六章 §2-(3)(4)）。`recipe.total_time_minutes` 可由各步 `duration_minutes` 求和得出，存了便于直接查询。

### 1.7 与技术方案 §4 / 需求文档的细化说明

本次落地过程中的几处细化（不冲突、仅补充）：

1. **`order.order_date + daily_seq` 拆分**：技术方案 §4.2 说「下单日期 + 当日编号联合唯一索引」，本设计拆为 DATE + INT 两个字段，便于编号生成与查询；display_no（如「0629-001」）不存字段、由两字段派生（单一可信源）
2. **`ingredient` 表合并原料档案与当前库存**：需求文档第五章 §2 是档案字段、§3 是库存字段，技术方案 §4.2 也是合并设计，本设计沿用——`current_stock` 是当前值、`ingredient_stock_log` 是变动留痕，二者配合
3. **`bot_conversation_log.payload` 用 JSONB 不分子表**：5 类痕迹结构差异大、JSONB 灵活度高；如未来查询性能成问题再考虑拆表
4. **`order_status_log.action` 用枚举**：技术方案 §4.2 只说「触发动作」，本设计细化为 8 种枚举（CREATE/UPDATE/START_COOKING/FINISH/MARK_DISPATCHED/CLOSE_PICKUP/CLOSE_DELIVERY/CANCEL）。**8 个枚举值与技术方案 §2.2 order 模块的 service 方法一一对应**：CREATE → `create.py`、UPDATE → `update.py`、5 个状态推进（START_COOKING/FINISH/MARK_DISPATCHED/CLOSE_PICKUP/CLOSE_DELIVERY）→ `lifecycle.py`、CANCEL → `cancel.py`；`query.py` 是查询不产生状态变更、不对应 action 枚举。便于按动作类型筛选与统计
5. **`api_token.token_hash` 加 UNIQUE 索引、`user.pin_hash` 不加**：技术方案 §4.2 已说明差异，本设计在索引层落地——token 校验是「客户端传明文 → 服务端 bcrypt.hash → SELECT 比对」可走索引，PIN 校验是「遍历 enabled 用户逐个 checkpw」无法走索引
6. **留痕表外键统一 `ON DELETE RESTRICT`**：技术方案未明确，本设计补充——即便关联业务记录被删，留痕必须保留可追溯

## 2. 订单状态机

订单是本系统状态最复杂的实体——7 个状态、8 个动作、3 种角色（店员 / 后厨 / 店主）参与的转换、不同状态下的库存联动差异。本节是 service 层（特别是 `lifecycle.py` / `cancel.py`）实现的权威依据。

原料、客户、配方等实体无真正意义上的状态机（数据为主、无状态流转），不在此节展开。

### 2.1 状态定义（7 个）

| 状态枚举 | 中文名 | 含义 | 进入条件 |
|---|---|---|---|
| `PENDING` | 待制作 | 订单已入库，等待后厨安排制作 | CREATE 动作 |
| `COOKING` | 制作中 | 后厨选择「现做」路径，正在按配方制作 | START_COOKING 动作 |
| `READY_FOR_PICKUP` | 待取 | 自取单蛋糕就绪，等待客户到店 | FINISH 动作（自取单） |
| `READY_FOR_DELIVERY` | 待配送 | 配送单蛋糕就绪，等待安排配送员 | FINISH 动作（配送单） |
| `DELIVERING` | 配送中 | 已交付配送员、送达途中 | MARK_DISPATCHED 动作 |
| `COMPLETED` | 已完成 | 客户取走 / 配送送达，订单终结 | CLOSE_PICKUP / CLOSE_DELIVERY 动作 |
| `CANCELLED` | 已取消 | 订单被取消，库存按既定策略处理 | CANCEL 动作 |

**终态**：`COMPLETED` 与 `CANCELLED` 是终态，不可再转换。

**第一期仅现做路径**：所有订单从 `PENDING` 经 `START_COOKING` 进入 `COOKING`，再经 `FINISH` 进入 `READY_FOR_*`。第二期补「从成品库存扣减」路径（`PENDING` → `READY_FOR_*`，不经过 `COOKING`），详见 §2.5。

**`CANCEL` 是横切动作**：从任意非终态都可触发（图中只画了几条代表性 CANCEL 箭头，实际覆盖 `PENDING` / `COOKING` / `READY_FOR_PICKUP` / `READY_FOR_DELIVERY` / `DELIVERING` 五种状态）。

**`UPDATE` 不改状态**：仅在 `PENDING` 状态内做字段修改，不切状态、不出现在转换图上。

### 2.3 转换规则表（8 个 ORDER_ACTION）

每个动作的「前置状态 / 目标状态 / 触发角色 / 联动动作」清单。**所有动作必须经 service 层事务**（更新 `order.status` + 写 `order_status_log` + 触发库存/WebSocket 副作用），禁止绕过 service 直接改 `status`。

#### 2.3.1 CREATE（创建订单）

| 项 | 内容 |
|---|---|
| 前置状态 | （订单不存在）|
| 目标状态 | `PENDING` |
| 触发角色 | 店员（机器人下单，主流路径）／ 店主、店员（后端管理页面手动补录，少数场景） |

| 状态约束 | 无 |

#### 2.3.2 UPDATE（改单，仅字段修改）

| 项 | 内容 |
|---|---|
| 前置状态 | `PENDING`（**唯一允许改单的状态**，需求文档第二章故事 5 §319） |
| 目标状态 | `PENDING`（状态不变） |
| 触发角色 | 店员（机器人改单）／ 店主、店员（后端管理页面） |
| service 入口 | `services/update.py` |

#### 2.3.3 START_COOKING（开始制作-现做路径）

| 项 | 内容 |
|---|---|
| 前置状态 | `PENDING` |

| 触发角色 | 后厨（后厨屏，需求文档第二章故事 6 §371 明确路径选择必须由后厨主动操作） |
| service 入口 | `services/lifecycle.py` |
| 联动动作 | (1) 锁定 `recipe_snapshot`（订单进入 COOKING 时锁定，按快照口径扣料）<br>(2) 填 `order.recipe_snapshot_id`<br>(3) 写 `order_status_log`（action=START_COOKING, old_status=PENDING, new_status=COOKING）<br>(4) WebSocket 推送状态变更到后厨屏与管理页面 |
| 状态约束 | 仅 `PENDING` 可触发；其他状态拒绝 |
| **第一期不实现** | 「从成品库存扣减」路径（`PENDING` → `READY_FOR_*`，不经过 `COOKING`）——第二期补，详见 §2.5 |

#### 2.3.4 FINISH（完工）

| 项 | 内容 |
|---|---|
| 前置状态 | `COOKING` |
| 目标状态 | `READY_FOR_PICKUP`（自取单）或 `READY_FOR_DELIVERY`（配送单），由 `order.pickup_method` 决定 |
| 触发角色 | 后厨（后厨屏，需求文档第六章故事 2） |
| service 入口 | `services/lifecycle.py` |
| 联动动作 | (1) 按 `recipe_snapshot` × `quantity` 扣减原料库存（写 `ingredient_stock_log`，source_type=FINISHED_GOODS_DEDUCT）<br>(2) **第一期简化口径**：成品诞生即视为被本订单即时匹配扣减——同一事务内完成「原料扣减 → 成品诞生 → 订单匹配」三步，不进 `finished_goods` 成品库存表、不写 `FINISHED_GOODS_RESTOCK` / `FINISHED_GOODS_MATCHED_DEDUCT` 两类流水（这两类流水第二期才区分，详见 §2.5 与 §1.2.5 STOCK_SOURCE「第二期扩展」段）<br>(3) 写 `order_status_log`（action=FINISH, old_status=COOKING, new_status=READY_FOR_*）<br>(4) WebSocket 推送 |
| 状态约束 | 仅 `COOKING` 可触发 |
| **第二期扩展** | 拆为「成品诞生扣减」+「订单匹配扣减」两阶段事务（需求文档第十三章第 4 节取舍 1）；本期简化为一步 |

|---|---|
| 前置状态 | `READY_FOR_DELIVERY`（配送单专属） |
| 目标状态 | `DELIVERING` |
| 触发角色 | 后厨或店员（HTTP 端：后厨屏 / 后端管理页面均可调用）+ 店员（MCP 端：机器人 `mark_dispatched` 工具）——需求文档第十章 §3 核心层 (4)e 明确「后厨或店员」均可触发 |
| service 入口 | `services/lifecycle.py`（HTTP 与 MCP 共用同一 service 方法） |
| 联动动作 | (1) 写 `order_status_log`（action=MARK_DISPATCHED, old_status=READY_FOR_DELIVERY, new_status=DELIVERING）<br>(2) WebSocket 推送 |
| 状态约束 | 仅 `READY_FOR_DELIVERY` 可触发（自取单永远进不了这个状态） |

#### 2.3.6 CLOSE_PICKUP（客户到店取货关闭）

| 项 | 内容 |
|---|---|
| 前置状态 | `READY_FOR_PICKUP`（自取单专属） |
| 目标状态 | `COMPLETED` |
| 触发角色 | 后厨（后厨屏，需求文档第二章故事 3 §221） |
| service 入口 | `services/lifecycle.py` |
| 联动动作 | 按订单当前 `payment_status` 分两个子场景：<br>**子场景 A（未付订单，UNPAID）**：合并按钮「已付款 / 已交付」——(1) 支付状态变更为 `PAID`、记录 `payment_method` 与收款时间<br>(2) 状态切换 `COMPLETED`<br>(3) 写 `order_status_log`（action=CLOSE_PICKUP, old_status=READY_FOR_PICKUP, new_status=COMPLETED, note 含支付方式）<br>**子场景 B（已付订单，PAID）**：仅交付按钮——(1) 仅状态切换 `COMPLETED`<br>(2) 记录交付时间<br>(3) 写 `order_status_log`（action=CLOSE_PICKUP, old_status=READY_FOR_PICKUP, new_status=COMPLETED）<br>支付状态保持 `PAID` 不变，**不得触发二次收款** |
| 状态约束 | 仅 `READY_FOR_PICKUP` 可触发；前端按钮可用性由 `payment_status` 决定（未付只能点合并按钮、已付只能点交付按钮） |

#### 2.3.7 CLOSE_DELIVERY（配送订单送达关闭）

| 项 | 内容 |
|---|---|
| 前置状态 | `DELIVERING`（配送单专属） |
| 目标状态 | `COMPLETED` |
| 触发角色 | 店员（机器人为主入口 `close_delivery_order` MCP 工具）；后端管理页面同步提供作为「机器人功能镜像」（技术方案 §5.2 line 482） |
| service 入口 | `services/lifecycle.py` |
| 联动动作 | 按订单当前 `payment_status` 分两个子场景：<br>**子场景 A（送达付款，UNPAID）**：(1) 支付状态变更为 `PAID`、记录 `payment_method` 与收款时间<br>(2) 状态切换 `COMPLETED`<br>(3) 写 `order_status_log`（action=CLOSE_DELIVERY, old_status=DELIVERING, new_status=COMPLETED, note 含支付方式）<br>**子场景 B（下单时已付，PAID）**：(1) 状态切换 `COMPLETED`<br>(2) 记录送达时间<br>(3) 写 `order_status_log`（action=CLOSE_DELIVERY, old_status=DELIVERING, new_status=COMPLETED）<br>支付状态保持 `PAID` 不变 |
| 状态约束 | 仅 `DELIVERING` 可触发；若订单在 `READY_FOR_DELIVERY` 状态（未发出）需走取消流程，不在本动作范围 |

#### 2.3.8 CANCEL（取消订单）

| 项 | 内容 |
|---|---|
| 前置状态 | 任意非终态（`PENDING` / `COOKING` / `READY_FOR_PICKUP` / `READY_FOR_DELIVERY` / `DELIVERING`） |
| 目标状态 | `CANCELLED` |
| 触发角色 | 店员（机器人为主入口）；后端管理页面同步提供 |
| service 入口 | `services/cancel.py` |
| 联动动作 | 按订单当前状态分**三组分支**（需求文档第二章故事 4 §277-285），每组分支内部步骤连续编号；同一 service 调用只命中其中一组：<br><br>**分支 A · `PENDING` / `COOKING`**（原料尚未扣减、成品未诞生）：<br>　(A1) 不触发库存动作<br>　(A2) 写 `order_status_log`（action=CANCEL, note 含取消原因）<br><br>**分支 B · `READY_FOR_PICKUP` / `READY_FOR_DELIVERY`**（蛋糕已成型、原料消耗已落账）：<br>　(B1) 处理蛋糕实物——**第一期只有「报损废弃」一种**：写一条 `ingredient_stock_log` source_type=LOSS 记录损耗（需求文档第十三章第 4 节取舍明确）<br>　(B2) 写 `order_status_log`（action=CANCEL, note 含「报损废弃」与取消原因）<br><br>**分支 C · `DELIVERING`**（蛋糕已交付配送员、物理上无法回滚）：<br>　(C1) 系统不自动处理库存<br>　(C2) 写 `order_status_log`（action=CANCEL, note 含协商结果——截回配送 / 折价 / 补做等由店员与客户线下协商后填入）<br><br>**三组分支共同收尾**：置 `order.status = CANCELLED`；WebSocket 推送状态变更到后厨屏与管理页面。详见 §4.1.5.1 取消订单弹窗的状态分支字段。 |
| 状态约束 | `COMPLETED` / `CANCELLED` 终态不可取消 |
| **配送员退回特殊情况** | `DELIVERING` 状态蛋糕被原样带回时，店员可在系统里手动选择「重新安排配送」（回到 `READY_FOR_DELIVERY`）、「客户自取」（回到 `READY_FOR_PICKUP`）或「入成品库存继续售卖」（按 `READY_FOR_*` 取消策略处理）——系统不自动判定退回原因。**第一期实现优先级低**，可作为第一刀切片之后的功能 |

### 2.4 状态约束矩阵（防误操作）

下表列出每个状态允许触发的动作集合，service 层应做前置校验、动作不允许时返回 400 + 业务错误码：

| 当前状态 | 允许的动作 |
|---|---|
| `PENDING` | UPDATE / START_COOKING / CANCEL |
| `COOKING` | FINISH / CANCEL |
| `READY_FOR_PICKUP` | CLOSE_PICKUP / CANCEL |
| `READY_FOR_DELIVERY` | MARK_DISPATCHED / CANCEL |
| `DELIVERING` | CLOSE_DELIVERY / CANCEL |
| `COMPLETED` | （无，终态） |
| `CANCELLED` | （无，终态） |

**注意**：`pickup_method` 进一步约束——自取单永远不会进入 `READY_FOR_DELIVERY` / `DELIVERING`，配送单永远不会进入 `READY_FOR_PICKUP`。`FINISH` 动作按 `pickup_method` 二选一决定目标状态。

## 3. API 契约概览

详细路径、请求体、响应体**不写在文档里**——FastAPI 的 Pydantic 模型在代码里定义、运行时自动生成 OpenAPI schema，前端用 `openapi-typescript` 同步生成 TS 类型，文档手抄一份会立刻过期。本节只写**接口设计的准则与约定**，让后端开发按准则实现、前端按约定调用，不需要每次为「字段叫什么」「错误码怎么命名」重新讨论。

### 3.1 契约生成与同步机制

```
backend/routers/*.py  ──Pydantic 模型──▶  FastAPI 运行时
                                              │
                                              ▼
                                      /openapi.json
                                              │
                          openapi-typescript  │  (前端 dev 期 npm script)
                                              ▼
                              frontend/src/api/types.gen.ts
                                              │
                                              ▼
                                  前端 import 与调用
```

- **单一可信源**：后端 Pydantic 模型。前端类型、OpenAPI 文档、`curl` 示例都从它派生
- **同步纪律**：任何 API 变更必须改 Pydantic 模型，commit 前跑 `npm run gen:api`（前端）拉取最新 `types.gen.ts`；CI 校验 `types.gen.ts` 与 `/openapi.json` 一致，不一致 fail build
- **contract-first 思想的实际落地（与技术方案 §5.1 一致，v1.0.13 同步对齐）**：技术方案 §5.1 提出 contract-first，采用「后端先行 + 真实联调」落地，**不另起独立 mock server**。具体流程：(1) 后端先写 Pydantic schema + router 注册 + service 层签名（业务逻辑可暂返回 stub 数据），跑起来 `/openapi.json` 即可用；(2) 前端 `npm run gen:api` 拉类型、直接调真实端点开发；(3) 后端逐步实现 service 业务逻辑、前端无感切换。理由：FastAPI 的 Pydantic → OpenAPI → TS types 链路天然就是 contract-first，独立维护 mock server 反而多一份与真实契约漂移的资产，对单人开发得不偿失
- **第一期范围声明**：本期无文件上传需求（菜单/配方/客户档案均无图片字段）；所有写操作单条提交，批量操作（如「一次性把多个待制作订单标记为开始制作」）第二期评估

### 3.2 路由模块化组织

承接技术方案 §2.2 的 `backend/app/modules/{module}/` 目录结构（简单模块扁平、复杂模块按行为拆子目录，规则见技术方案 §2.3 第 6 条）。每个模块的 router 文件承担：路径声明、Pydantic schema 定义、调 service 层、统一响应包装。**不写业务逻辑**——业务逻辑全在 service 层。

各模块的 router 归属（与 §1.1 表清单一一对应）：

### 3.3 Pydantic schema 分层约定

每个模块的 schemas 按「用途」分四类，文件命名 `schemas/{module}.py`：

| 后缀 | 用途 | 示例 |
|---|---|---|
| `*Create` | POST 请求体 | `OrderCreate`、`CustomerCreate` |
| `*Update` | PATCH 请求体（**所有字段 Optional**，未传 = 不变） | `OrderUpdate`、`IngredientUpdate` |
| `*Response` | 单条响应体 | `OrderResponse`、`CustomerResponse` |
| `*ListItem` | 列表项（**裁剪字段**，不含详情字段，避免列表接口臃肿） | `OrderListItem`、`CustomerListItem` |

**约定**：
- `*Response` 与 ORM 模型不强制 1:1——`OrderResponse` 应包含前端展示需要的全部字段（含关联的 `customer.wxid`、`menu.name` 等），用 `from_attributes=True` 配合 service 层 join 一次取齐
- `*Update` 全字段 Optional，service 层用 `model_dump(exclude_unset=True)` 区分「字段未传」与「字段传 null」（后者表示显式清空，例如改单时 `delivery_address = null`）
- 严禁用同一个 schema 同时承担 Create 和 Update——字段必填性语义不同，混用必有 bug

### 3.4 通用响应包装

所有 HTTP API 与 MCP 工具统一返回：

```python
class ApiResponse(BaseModel, Generic[T]):
    data: T | None = None
    error: ApiError | None = None

class ApiError(BaseModel):
    code: str          # 大写下划线，机器可读
    message: str       # 中文，给操作者看
    details: dict = {} # 可选，结构化补充（如缺料列表）
```

**纪律**：
- 成功：`data` 必填、`error = null`、HTTP 200
- 业务失败：`data = null`、`error.code` 是机器可读错误码、HTTP 400/404/409
- 鉴权失败：HTTP 401（未登录 / session 失效 / Bearer Token 无效）/ 403（已登录但角色不够）/ 429（PIN 锁定或限流）
- 系统错误：HTTP 500 + `INTERNAL_ERROR`，堆栈写 structlog、不暴露给前端

### 3.5 列表接口的 query 参数约定

所有列表接口（`GET /api/orders`、`GET /api/customers` 等）统一约定：

| 参数 | 类型 | 说明 |
|---|---|---|
| `page` | int | 页码，从 1 开始，默认 1 |
| `page_size` | int | 每页条数，默认 20，最大 100（防前端误传大值拖垮 DB） |

```python
    page_size: int
```

### 3.6 错误码命名规范

错误码分两层，避免与技术方案 §5.4 已有命名冲突：

**第一层 · 鉴权与系统级（沿用技术方案 §5.4 已定）**：HTTP 401/403/429/500 对应的兜底错误码，前端按 HTTP 状态码即可路由处理：

| HTTP 状态码 | 错误码 | 含义 |
|---|---|---|
| 401 | `UNAUTHORIZED` | 未登录 / session 失效 / Bearer Token 无效 |
| 403 | `FORBIDDEN` | 已登录但角色无权限 |
| 429 | `RATE_LIMITED` | PIN 锁定或 API 限流 |
| 500 | `INTERNAL_ERROR` | 系统错误，细节写日志不暴露 |

**第二层 · 业务错误（本文档新增命名规范）**：模块前缀错误码（机器可读字符串），调用方按 `error.code` 路由具体业务逻辑。传输载体分两类——HTTP API 端点走 HTTP 400 + JSON `{error: {code, message, details}}`，MCP 工具走 MCP 协议的 `isError: true` + 结构化 payload（MCP 不复用 HTTP 状态码，见技术方案 §3.3）。**关键：同一 code 走哪个载体由调用入口决定、不由前缀决定**——组 B 的 `STOCK_INSUFFICIENT` / `MENU_OFFLINE` 等 code 在 HTTP API 入口（后端管理页面 / 后厨屏）走 HTTP 400、在 `create_order` MCP 工具入口（机器人）走 MCP 结构；只有 `BOT_*` 是入口特性独占、永远不会出现在 HTTP API（因为后端管理页面与后厨屏是结构化表单输入、不会产生消息解析类错误）。前缀按**语义维度**分两组，避免维度混杂。

**组 A · 横切/入口类前缀**（与具体业务实体无关，覆盖鉴权、入口特性等横切关注点。组 A 内部可继续细分入口类（`BOT_*`）vs 横切类（`AUTH_*`），但本期前缀少（仅 2 个）暂不拆第二层；未来扩到 4-5 个前缀时再评估是否拆，避免重蹈维度混杂覆辙）：

| 前缀 | 含义 | 示例 |
|---|---|---|
| `AUTH_*` | 鉴权细节错误（与 §5.4 第一层的 UNAUTHORIZED/FORBIDDEN 互补——后者按 HTTP 状态码路由、前者按细分原因路由） | `AUTH_INVALID_PIN`（PIN 校验失败但未到锁定阈值）、`AUTH_LOCKED`（PIN 锁定，detail 含剩余锁定时长） |
| `BOT_*` | 机器人入口独有错误（消息解析、对话流程异常；后厨屏与后端管理页面是结构化表单输入、不会出现这类错误）。**传输载体是 MCP 工具返回结构**（`isError: true` + payload），**不走 HTTP API、不绑定 HTTP 状态码**——MCP 协议本身不携带 HTTP 状态码（见技术方案 §3.3） | `BOT_MESSAGE_INCOMPLETE`（消息缺字段、店员需补全重发）、`BOT_CUSTOMER_ARCHIVE_MISS`（档案不存在但本流程要求存在） |

**组 B · 业务实体类前缀**（与具体业务实体的状态/数据有关）：

| 前缀 | 含义 | 示例 |
|---|---|---|
| `ORDER_*` | 订单业务错误 | `ORDER_NOT_FOUND`、`ORDER_STATE_FORBIDDEN`（状态约束违反，detail 含当前状态与允许动作集合）、`ORDER_DAILY_SEQ_CONFLICT` |
| `STOCK_*` | 库存业务错误 | `STOCK_INSUFFICIENT`（缺料，detail 含原料清单）、`STOCK_LOG_FORBIDDEN` |
| `CUSTOMER_*` | 客户档案业务错误 | `CUSTOMER_NOT_FOUND`、`CUSTOMER_PHONE_DUPLICATE` |
| `MENU_*` | 菜单业务错误 | `MENU_NOT_FOUND`、`MENU_OFFLINE`（商品已下架） |
| `RECIPE_*` | 配方业务错误 | `RECIPE_NOT_FOUND`、`RECIPE_REFERENCED_IN_PENDING_ORDERS` |

**前缀归属判断纪律**（开发期新增错误码时按此判断归哪组）：

1. 先问「这是哪个业务实体的事？」——能归到业务实体就用组 B 前缀（如「订单状态不允许操作」归 `ORDER_*`、「商品已下架」归 `MENU_*`）
2. 只有跨业务实体或与业务实体无关的错误才归组 A——如「鉴权细节」（`AUTH_*`）、「机器人消息解析」（`BOT_*`）
3. **禁止把业务错误硬塞到入口前缀**：如「机器人下单时库存不足」本质是 `STOCK_INSUFFICIENT`，不要因为「发生在机器人入口」就改成 `BOT_*`——入口是触发场景、不是错误归属
4. **禁止把入口特性错误硬塞到业务前缀**：如「机器人消息缺字段」本质不是订单错误（订单都还没创建），归 `BOT_MESSAGE_INCOMPLETE` 而非 `ORDER_*`
5. **同一业务实体的错误在不同入口有不同致命性时，通过 `details` 字段表达、不通过换前缀表达**：如「订单状态不允许操作」在后厨屏与后端管理页面都是可重试的业务提示（`ORDER_STATE_FORBIDDEN`、HTTP 400），不因「发生在后厨屏」就改成 `KITCHEN_ORDER_STATE_FORBIDDEN`——前缀归业务实体、致命性差异走 `details`，避免回到「入口即前缀」的维度混杂

**纪律**：
- 业务错误必须落在 400，不得复用 401/403/429（避免前端鉴权拦截器误判）
- `AUTH_INVALID_PIN` 与 `AUTH_LOCKED` 仍走 HTTP 401 / 429（沿用技术方案 §5.4 的鉴权 HTTP 状态码），但 `error.code` 区分细分原因
- `BOT_*` 错误仅在 MCP 工具返回结构中出现（`isError: true` + payload），MCP 协议本身不携带 HTTP 状态码（见技术方案 §3.3 line 297-298），不存在「BOT_* 走 HTTP 400」一说；后端管理页面与后厨屏本期不会产生 `BOT_*` 错误（结构化表单输入），若未来这两个入口出现类比入口特性错误，再单独评估走 HTTP 400 还是新增 `FORM_*` 之类前缀
- 错误码集中在 `backend/app/constants/error_codes.py` 维护，新增错误码必须在这里登记，避免散落在各 router
- 新增前缀（如未来加 `ADMIN_*` / `KITCHEN_*` 等入口前缀，或 `MIGRATION_*` 等横切前缀）必须先在本节登记并说明归属哪一组

### 3.7 WebSocket 事件协议

后厨屏订阅 `wss://.../api/kitchen/ws`（session cookie 鉴权，连接时校验失败立即 close）。后端推送的事件类型：

| event | 触发时机 | payload |
|---|---|---|
| `order.created` | CREATE 动作完成 | 完整订单（OrderListItem） |
| `order.updated` | UPDATE 动作完成 | 订单 id + change_diff |
| `order.state_changed` | 状态推进（START_COOKING / FINISH / MARK_DISPATCHED / CLOSE_*） | 订单 id + old_status + new_status |
| `order.cancelled` | CANCEL 动作完成 | 订单 id + 取消原因 |

**协议约定**：
- 每条消息为 JSON：`{ "event": "order.created", "payload": {...}, "ts": "2026-06-29T10:00:00Z" }`
- 前端连接后发送 `{ "event": "subscribe", "payload": { "client_type": "kitchen" } }` 完成握手；后端只对 `client_type=kitchen` 的连接推流（防止管理页面误连）
- 心跳：前端每 30s 发 `{ "event": "ping" }`，后端 60s 无心跳断开
- 重连：前端指数退避（1s → 2s → 4s → ... 上限 30s），重连成功后主动调一次 `GET /api/kitchen/today` 拉取期间错过的更新（不依赖 WebSocket replay）
- 多 worker 部署：技术方案 §8 决策——Uvicorn 单 worker，本期不上 Redis Pub/Sub

### 3.8 内部回写端点（`/api/internal/*`）

仅供 OpenClaw Plugin 等程序访问，**不接受 session cookie、只接受 Bearer Token**：

- `POST /api/internal/bot-conversation-log`：Plugin 回写对话留痕（session_id / step_type / payload / operator_wxid / order_id）

Caddy 路径过滤层面 `internal` 段对内网/同机放开、对外不暴露（如果 OpenClaw 与 backend 同机，可绑定 127.0.0.1；如果分机，靠 Bearer Token 鉴权）。

### 3.9 缺料检测返回结构（`ShortageResult`）

`modules/ingredient/services/shortage_check.py` 的 service 返回类型，同时被订单创建 service、改单 service、`parse_order_message` MCP 工具、`check_inventory_shortage` MCP 工具复用。该结构非 DB 字段、非 JSONB 列，是 service 层与 API/MCP 层之间的契约，集中定义在 §3 避免每个调用方各自脑补字段：

```python
class ShortageItem(BaseModel):
    ingredient_id: int          # 短缺原料 id
    ingredient_name: str        # 原料名（便于回执直接展示，免得调用方再 join）
    unit: str                   # 单位（kg / 个 / ml 等，与 ingredient.unit 一致）
    required_quantity: float    # 本次订单需求量（recipe 用量 × order.quantity）
    committed_quantity: float   # 「待制作 + 制作中」订单对该原料的承诺量合计
    current_stock: float        # ingredient.current_stock 当前库存余额
    projected_balance_after_deduction: float
        # 预计扣减后余额 = current_stock - committed_quantity - required_quantity
        # 负数即代表本次接单后会缺料；正数代表库存充足
    shortage_quantity: float    # 缺口绝对值（max(0, -projected_balance_after_deduction)）

class ShortageResult(BaseModel):
    is_shortage: bool                  # 是否缺料（missing_items 非空时为 True）
    missing_items: list[ShortageItem]  # 缺料原料清单（按 ingredient_id 升序）
    checked_at: datetime               # 检测时间戳（承诺量是时点快照、便于排查）
```

**字段语义约定**：
- `projected_balance_after_deduction` 是关键字段——它表达了「假设本单接下、且所有已接未完工订单都按计划扣料后，库存会变成多少」。负值就是缺口。
- `committed_quantity` 严格按 §2.3.1 联动口径计算——只统计 `status IN (PENDING, COOKING)` 的订单（这两个状态的原料尚未实际扣减，FINISH 状态推进后才扣），避免重复计算
- 调用方拿到 `ShortageResult` 后**自行决定如何呈现**：机器人回执把它转成自然语言（「蓝莓还差 0.3kg」），管理页面改单弹窗把它做成表格预览，service 本身不关心展示形态
- 该结构**永不入库**——是计算结果、不是持久化数据，每次调用即时算

#### 4.3.1 API 客户端封装

```typescript
// frontend/admin/src/api/client.ts（kitchen 应用同样一份）
const client = axios.create({ baseURL: '/api', withCredentials: true })
```

## 5. 测试策略

### 5.0 设计哲学（先于工具选型）

本系统功能复杂度集中在「订单状态机 + 三入口协作 + 客户档案异步补偿 + 机器人对话留痕」——单测能验证 service 层逻辑正确性，但**无法验证意图**（"店员说一句话 → 后厨屏亮起 → 客户取走蛋糕 → 库存正确扣减"这条链路只能 E2E 验证）。

测试设计的核心原则（承接 CLAUDE.md「真实验证准则」）：

1. **意图优于实现**：每个用户故事（需求文档第二/三/五/六章）至少有一个 E2E 测试覆盖，单测可以全绿但 E2E 红 → 系统就是坏的
2. **Fail loud**：任何「完成」声明必须显式列出跳过了什么 / 有什么 warning / 没验证到什么；最贵的失败是看起来像成功的失败
3. **三入口独立可测**：admin（管理页面）/ kitchen（后厨屏）/ bot（机器人）三个入口的 E2E 互不依赖，任何一个入口出问题不影响另两个入口的测试运行
4. **测试数据可重建**：每个 E2E 测试从干净数据库开始、跑完销毁，测试之间无副作用依赖
5. **真实链路、stub 外部依赖**：跑真实 backend + 真实 DB + 真实浏览器，但 stub 掉真实 LLM（贵且不稳定）、真实微信（不可能自动化）、真实推送通道。**外部接入层（Caddy 的 TLS / OpenClaw 的 LLM）的端到端验证归 §5.7 真实运行验证**——E2E 自动化测试不重复起 Caddy，直连 backend:8000 不损失业务覆盖（理由见 §5.5）

### 5.1 测试金字塔与职责分层

```
                    ┌──────────────┐
                    │   E2E 测试    │   Playwright + 真实浏览器
                    │   ~30 场景    │   + 真实 backend + 真实 DB
                    └──────┬───────┘
                  ┌────────┴────────┐
                  │  集成测试        │   pytest + TestClient
                  │  ~80 用例        │   + 真实 DB（独立 schema）
                  └────────┬────────┘
                ┌──────────┴──────────┐
                │    单元测试           │   pytest，纯函数 / service 层
                │    ~200 用例          │   mock DB 与外部依赖
                └─────────────────────┘
```

各层职责：

| 层 | 覆盖目标 | 不覆盖 | 工具 |
|---|---|---|---|
| **单元** | service 层业务逻辑、状态机校验、缺料检测计算、change_diff 生成、价格重算、订单编号生成、库存余额计算 | DB 真实读写、HTTP 层、WebSocket、跨模块协作 | pytest + pytest-mock |
| **集成** | router → service → DB 全链路、SQLAlchemy ORM 与 PG 行为、Alembic 迁移可重放、Pydantic 校验、鉴权 dependency、事务隔离 | 浏览器渲染、WebSocket 客户端、机器人 MCP 调用、真实 LLM | pytest + httpx (AsyncClient) + 真 PG 容器 |
| **E2E** | 用户视角的端到端场景、三入口协作、WebSocket 推送时序、浏览器交互（PIN 输入、按钮点击、卡片切换） | 内部实现细节（单测覆盖） | Playwright + 真实浏览器 + 真 backend 容器 |

**关键纪律**：单元测试不能跨模块（不调其他 service）；集成测试不能调外部 HTTP（不调 OpenClaw）；E2E 测试不 mock 后端（不调真实 LLM 但调真实 backend）。任何一层越界都意味着测试金字塔变形。

**跨 service 调用的分层归属**（边界细化）：当订单 service 在创建订单时调用客户档案 service 拉取历史备注、或调原料 service 跑缺料检测——这种「同进程内跨模块」调用归**集成测试**层（连真 DB、跑真实 service 间调用链路），不算单元测试。单测严格只测「单个 service 文件内的纯函数 / 类方法 + mock 掉本 service 之外的一切」。

### 5.2 工具选型

| 用途 | 工具 | 选型理由 |
|---|---|---|
| 单元/集成测试 | pytest + pytest-asyncio + httpx | Python 生态事实标准；httpx 的 AsyncClient 与 FastAPI 的 TestClient 协议一致 |
| 测试 DB | Postgres 15 容器（独立于开发 DB） | 与生产同构；docker-compose.test.yml 起 |
| 数据库 fixture | pytest-postgresql 或自建 fixture（推荐后者，灵活） | 自建 fixture 可控地 truncate + seed |
| E2E 浏览器测试 | Playwright（Python 版） | 跨浏览器、原生支持 WebSocket、自动等待、trace viewer（失败时回放录屏与每一步 DOM 快照，定位用）调试体验好；优于 Cypress（Cypress 对多 tab/多 origin 支持弱） |
| E2E 浏览器 | Chromium（Playwright 默认） | 不需要测 Firefox/Safari 兼容性（小店场景只跑 Chromium） |
| MCP 调用模拟 | 直连 backend 的 `/mcp/*` 端点 + httpx 流式 client | E2E 不调真实 OpenClaw，直接打 backend 的 MCP server |
| LLM 响应模拟 | fixture 文件（按场景固定的 LLM 输出） | 不调真实 LLM——稳定性差、成本高、与系统行为验证无关 |
| 测试报告 | pytest-html + Playwright HTML report + Allure（可选） | CI 上看趋势用 Allure，本地调试用各自原生报告 |

### 5.3 E2E 场景清单（按需求文档用户故事映射）

每个场景对应一个或多个需求文档用户故事，命名规范 `e2e/tests/test_<entry>_<story>.py`，entry ∈ {`admin`, `kitchen`, `bot`}。

#### 5.3.1 机器人入口 E2E（核心，覆盖订单生命周期主要路径）

| 场景 ID | 对应需求文档故事 | 描述 | 关键断言 |
|---|---|---|---|
| `bot_order_create_happy` | 第二章故事 1 | 店员发完整消息 → 机器人回执 → 店员确认 → 订单入库 → 后厨屏 WebSocket 收到推送 | 回执 5s 内返回；订单编号格式 `MMDD-NNN`；DB 出现订单；后厨屏页面 DOM 出现新卡片；提示音播放事件触发 |
| `bot_order_create_paid` | 第二章故事 1-(3)-(1) | 消息附带「已付 微信转账」→ 回执显示支付状态 → 入库时支付状态与方式落账 | order.payment_status = PAID；order.payment_method = WECHAT |
| `bot_order_create_unpaid` | 第二章故事 1-(3)-(2) | 消息不附带支付信息 → 默认未付 → 回执显式展示「未付」 | order.payment_status = UNPAID；回执文本含「未付」 |
| `bot_order_create_with_shortage` | 第五章故事 2 | 库存不足 → 缺料检测 → 回执标记缺料 → 店员回复「仍然接单」→ 入库 | 回执含「原料蓝莓可用量不足」；订单入库；不扣库存（current_stock 不变） |

### 5.4 测试数据 fixtures 与 seed 策略

测试数据分两层：

**基础 seed（`tests/seed/base.sql` 或 `conftest.py` 中的 seed 函数）**：每个测试 session 开始时灌一次，覆盖整个系统最小可运行数据集：

- 1 个店主 user（PIN = `000000`，role = OWNER）
- 1 个店员 user（PIN = `111111`，role = STAFF）
- 1 个后厨 user（PIN = `222222`，role = KITCHEN）
- 1 个 api_token（Bearer Token，给 bot simulator 用）
- 5 个 menu（覆盖在售/下架、有配方/无配方、不同保质期）
- 8 个 ingredient（覆盖充足/低于安全线/已欠账）
- 3 个 recipe（对应 3 个在售 menu；含 steps JSONB）
- 3 个 customer（含电话/地址/备注三类关联；备注覆盖「过敏」「偏好」「中性」三类）

**场景级 fixture（`tests/fixtures/<scenario>.json`）**：每个 E2E 场景的输入数据 + LLM 响应 stub：

```json
{
  "scenario_id": "bot_order_create_happy",
  "llm_responses": [
    {"input": "wxid_abc123 明天下午 3 点要个蓝莓蛋糕", "output": {"tool": "create_order", "args": {...}}},
    {"input": "对", "output": {"tool": "confirm_order", "args": {...}}}
  ],
  "expected_db_state": {...},
  "expected_bot_reply_contains": ["蓝莓蛋糕", "明天", "15:00"]
}
```

**纪律**：
- 每个测试**用例**开始前 truncate 业务表（保留 user 与 menu 等基础 seed），保证用例间无副作用
- 测试**绝不能**依赖运行顺序
- LLM 响应 fixture 按场景固定，跑 E2E 时按 input 匹配 output，**不调真实 LLM**
- 时间相关测试用 `freezegun`（Python 时间冻结库，让时间相关测试可重放）冻结时间，避免跨日订单编号生成 flaky

**未单独 seed 的可观察项**：「采购后冲减欠账」状态（current_stock 从负数被采购冲回正数）不在基础 seed 中预置——理由是这是一个**动作结果**而非静态状态，由 `admin_purchase_register` 场景动态触发并验证；如未来发现冲减逻辑有边界 bug（如冲减顺序错乱），可在该场景内补一个「先报损扣到负、再采购冲减」的子用例。

### 5.5 测试环境

`docker-compose.test.yml`（与生产 `docker-compose.yml` 分离）：

```yaml
services:
  test-db:           # 独立 PG 容器，与开发 DB 物理隔离
    image: postgres:15
    environment:
      POSTGRES_DB: cake_test
    tmpfs: /var/lib/postgresql/data  # 内存盘，跑完销毁、不留痕迹

  test-backend:      # 真实 backend 镜像，连测试 DB
    build: ./backend
    environment:
      DATABASE_URL: postgresql://...@test-db/cake_test
      TESTING: "true"
    depends_on: [test-db]

  # 不起 Caddy：E2E 直接打 test-backend:8000，TLS 在 E2E 测试中无业务意义
  # 不起 OpenClaw：bot simulator 直接调 /mcp/* 端点

  playwright-runner:  # E2E 测试运行器
    image: mcr.microsoft.com/playwright/python:v1.42.0-jammy
    depends_on: [test-backend]
    volumes:
      - ./e2e:/e2e
      - ./playwright-report:/report
```

**关键决策**：
- 测试 DB 用 tmpfs（内存盘），跑完即销毁；不污染开发 DB
- 不起 Caddy：Caddy 只做 TLS 与路径过滤，E2E 直连 backend 不损失业务覆盖；TLS 测试（证书校验）作为单独的运维测试，不在功能 E2E 范围
- 不起 OpenClaw：bot simulator 直接调 MCP，覆盖业务路径；OpenClaw 与 LLM 的集成作为「真实验证」环节（详见 §5.7）
- 测试容器用 `cake_test` 库，与开发 `cake_dev`、生产 `cake_prod` 完全隔离

### 5.7 真实运行验证（CLAUDE.md 强制，每阶段 ship 前必须跑）

E2E 自动化测试覆盖了「意图」层，但有几类东西**自动化测不了**，必须人工 + 真实环境跑一次：

| 验证项 | 方式 | 跑的时机 |
|---|---|---|
| **真实 OpenClaw + 真实 LLM 接单** | 店主用真实微信发消息 → OpenClaw 调真实 backend → 看回执 | 第一刀切片 ship 前 + 每次大改 MCP 工具签名后 |
| **真实平板后厨屏** | 把平板挂墙上、跑 24 小时、看 WebSocket 是否稳定、息屏是否真的不发生 | 第一刀切片 ship 前 |
| **真实 Caddy + 自签证书** | 客户端设备导入 CA、访问 https://cake-shop.local | 第一次部署到测试环境时 |
| **跨日订单编号** | 改系统时间或等到真跨日、跑订单、验证编号重置 | 上线前一周内做一次 |
| **PIN 锁定 10 分钟真实计时** | 错 5 次 → 等真实 10 分钟 → 验证解锁 | 上线前做一次（自动化测试用 freezegun mock） |
| **客户档案备注异步补偿失败 → 仅写日志** | 故意把 customer_note 写挂掉、下一单、看 structlog WARN | 第一刀切片 ship 前 |

**未列出的可观察项及理由**：
- **多 worker 部署**：技术方案 §8 决策 3 明确「Uvicorn 单 worker 部署、多 worker 是 YAGNI」，本期不上、无需真实运行验证
- **Alembic 迁移在生产数据上的回滚**：技术方案 §8 决策 6 已定「所有迁移前向兼容、删字段三步走」，回滚不被推荐；上线前只验「正向迁移可在生产数据上跑通」（属技术方案 §7.3 tag v* 流水线范畴），不验回滚
- **配方快照跨版本兼容**：`kitchen_recipe_snapshot` E2E 已覆盖「订单进入 COOKING 时锁定快照、之后配方修改不影响订单」的核心断言；「跨多个版本（v1 → v2 → v3）的极端兼容性」不上真实运行验证清单，理由是 schema 设计已保证不可变性——`recipe_snapshot.content` 是 JSONB 全量快照（含原料清单、步骤、损耗率），生成后不再读 `recipe` / `recipe_ingredient` 表，故源表后续任何变更都不会污染历史订单口径

**真实运行验证的证据要求**（承接 CLAUDE.md Fail loud）：
- 贴截图 / 录屏 / 命令输出，不接受「我跑过了」
- 显式列出跳过了什么（如「Caddy TLS 跨网段验证跳过，理由是单店内网」）
- 显式列出未验证项（如「真实并发下单编号冲突只跑了 10 并发，更高并发未验证」）

### 5.8 ship 前必跑 E2E 清单

每次合并到 main 分支前，以下 E2E 必须全绿（CI 自动执行）：

1. **核心订单生命周期**（4 个）：`bot_order_create_happy` + `kitchen_board_state_advance` + `cross_bot_to_kitchen` + `bot_close_delivery_paid`
2. **状态机约束**（3 个）：`bot_order_update_state_forbidden` + `bot_order_cancel_pending` + `bot_order_cancel_ready`
3. **鉴权**（3 个）：`kitchen_login_pin` + `kitchen_login_lockout` + `admin_login`
4. **跨入口协作**（2 个）：`cross_bot_to_kitchen` + `cross_admin_update_kitchen_reflect`
5. **数据完整性**（2 个）：`bot_conversation_log_persisted` + `admin_purchase_register`
6. **WebSocket 时序**（2 个）：`kitchen_websocket_reconnect` + `kitchen_board_columns`

合计 16 个 E2E 必跑。其他 ~14 个 E2E 场景作为「完整套件」每周末跑一次（GitHub Actions schedule）。

### 5.9 CI 集成（GitHub Actions）

本节是对技术方案 §7.3 CI/CD 流水线的**测试编排细化**——§7.3 只列了 push to main（lint+pytest+构建）与 tag v*（部署）两条主线，本节补上 PR-gate、E2E job、schedule 三项测试相关编排。落地时合并到同一份 `.github/workflows/test.yml`，不另起 workflow。

```yaml
# .github/workflows/test.yml（示意）
jobs:
  unit-test:
    runs-on: ubuntu-latest
    steps: [setup-python, pip install, pytest tests/unit/]
    # < 30s

  integration-test:
    runs-on: ubuntu-latest
    services: { postgres: ... }
    steps: [setup-python, pip install, pytest tests/integration/]
    # < 2 min

  e2e-test:
    runs-on: ubuntu-latest
    steps:
      - setup-docker-compose
      - docker-compose -f docker-compose.test.yml up -d
      - wait-for-backend-healthy
      - playwright-install
      - pytest e2e/ --browser chromium
      - upload-artifact: playwright-report
    # 5-10 min

  pre-merge-gate:
    needs: [unit-test, integration-test, e2e-test]
    # 三个 job 都绿才能合并 PR
```

**CI 纪律**：
- 单元 + 集成 + ship 前 16 个核心 E2E：每个 PR 必跑，全绿才能合并
- 完整 E2E 套件（~30 个）：每周末 schedule 跑一次，结果贴 Slack/邮件
- Playwright 报告 artifact 保留 30 天，便于回溯失败截图与 trace

### 5.10 测试金字塔反模式（禁令）

- **禁止单测 mock 一切**：单测可以 mock DB 与外部依赖，但不能 mock 同模块的 service——mock 同模块等于没测
- **禁止集成测试不连真 DB**：用 SQLite 跑集成测试等于没测（PG 与 SQLite 的 JSONB、事务隔离、CHECK 约束行为差异巨大）
- **禁止 E2E mock backend**：E2E 必须连真实 backend，mock backend 等于把 E2E 降级为单测，浪费运行成本
- **禁止 E2E 调真实 LLM**：LLM 输出非确定性，E2E 必须 fixture stub；真实 LLM 验证归 §5.7 真实运行验证
- **禁止测试间共享状态**：每个测试用例从干净状态开始，不允许「跑完用例 A 后跑用例 B 复用 A 的数据」
- **禁止跳过失败的测试**：CI 上有测试红 → 修，不允许 `@pytest.mark.skip` 蒙混过关；只有「该测试场景已不适用」（如功能下线）才能删测试

## 6. 第一刀切片任务分解

### 6.1 第一刀的范围与边界

「第一刀」是项目从「只有文档」到「能跑起来一个最小可工作链路」的跨越。**第一刀的目标是验证架构与核心链路**，不是交付完整功能。

**第一刀包含**（店员通过机器人下单 happy path 全链路）：

- 店员在 OpenClaw 发消息（含 wxid + 客户消息文本）→ OpenClaw 调 `query_customer_archive` MCP 工具拉客户档案 → OpenClaw 调 `parse_order_message` MCP 工具识别订单字段（内部调 `check_inventory_shortage` 算缺料）→ 回执给店员确认 → 店员回复「对」→ OpenClaw 调 `create_order` MCP 工具入库 → 订单写入 `order` 表（status=PENDING）+ `order_status_log` CREATE + `bot_conversation_log` 五类痕迹 + 缺料检测结果随回执返回 + 客户档案备注异步补偿
- MCP 工具命名严格按技术方案 §3.2 单一可信源：`parse_order_message`（解析）/ `query_customer_archive`（查档案）/ `check_inventory_shortage`（缺料）/ `create_order`（入库）四个工具，**不引入 `confirm_order` 等额外工具**——「识别后让店员确认」的交互在 OpenClaw 侧通过 prompt + 回执 + 二次 `create_order` 调用完成，不需要后端拆两步

**第一刀不包含**（明确推迟到第二刀及之后）：

- 后厨屏（订单入库后停在 PENDING，无人推进状态）
- 改单 / 取消（订单状态机只实现 CREATE，UPDATE / CANCEL / 状态推进全推迟）
- 机器人追问（先做单轮接单——消息完整则直接识别、消息缺字段则返回错误；多轮追问推迟）
- WebSocket 推送（无后厨屏，推送无消费者）
- 后端管理页面（暂用 SQL 查表确认数据）
- 成品库存、现货销售、数据看板（第二期整体）

**第一刀的简化决策**（不等于偷懒，而是验证架构所需的最小集）：

| 项 | 第一刀做法 | 第二刀及之后 |
|---|---|---|
| 缺料检测 | 完整实现（业主关心，且与 §5 故事 2 验收标准对齐） | 不变 |
| 客户档案拉取 | `query_customer_archive` MCP 工具按 wxid 查档案；档案存在则回填电话/地址，不存在则只入 wxid | 不变 |
| 客户档案备注异步补偿 | 完整实现（FastAPI BackgroundTasks + 重试 3 次 + 失败仅写日志） | 不变 |
| 订单编号生成 | 完整实现（按需求文档第二章 §4 规则） | 不变 |
| 多轮追问 | **简化**：缺字段时返回 `BOT_MESSAGE_INCOMPLETE` 错误码 + 提示店员补全消息重发 | 实现完整追问（多轮对话） |
| 支付方式识别 | 完整实现（已付/未付，§2.3.1 CREATE 联动） | 不变 |
| 状态推进 | **不实现**（订单停 PENDING） | 第二刀做完整状态机 |
| 改单 | **不实现** | 第二刀做 |
| 取消 | **不实现** | 第二刀做 |

### 6.3 第一刀的验收标准

按 CLAUDE.md「真实验证准则」+「Fail loud」双重准则，第一刀 ship 前必须显式满足：

1. **集成测试全绿**：`pytest tests/integration/` 全部通过，覆盖率 ≥ 70%
2. **E2E 测试跑通**：`bot_order_create_happy` 场景 Playwright 跑通、报告留档
3. **真实运行验证留档**：店主用真实微信发消息 → 完整跑通接单 → 截图 + OpenClaw 日志 + backend 日志贴到 PR 描述
4. **Fail loud 清单**：在 PR 描述显式列出：
   - 跳过了什么（如「后厨屏不做、WebSocket 推送不验证」「多轮追问不实现」）
   - 有什么 warning（如 alembic 迁移警告、依赖 deprecation）
   - 没验证到什么（如「真实并发只跑了 10 并发订单号生成，更高并发未验证」）
5. **文档同步**：第一刀开发过程中如发现设计文档（技术方案 / 详细设计 / 需求文档）与实际实现不一致，**必须**在 PR 中同步修订文档（CLAUDE.md 红线）

**第一刀完成 ≠ 系统可用**——只是验证了「架构跑得通」。后续阶段 C/D/E 才能让系统真正可用。

- **每阶段 ship 前必须跑 §5.8 ship 前 16 个核心 E2E**（阶段 C 之后；阶段 B 第一刀 ship 时 E2E 套件还不完整，跑 T18 + 集成测试即可）
