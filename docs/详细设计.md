### 0.3 与技术方案、需求文档的关系

### 0.4 设计约定

**命名**
- 表名 / 字段名：snake_case
- 主键：统一用 `<表名单数>_id`（如 `order_id`、`customer_id`），自增 BIGINT
- 外键：`<引用表单数>_id`
- 时间字段：`*_at` 后缀（`created_at`、`updated_at`、`last_login_at`），类型 TIMESTAMPTZ
- 布尔字段：`is_*` 或 `has_*` 前缀
- 枚举字段：PostgreSQL ENUM 类型，值大写下划线（如 `ORDER_STATUS.PENDING`）

**类型缩写**
- `BIGINT`：64 位整数（主键、外键）
- `INT`：32 位整数（小范围计数）
- `TEXT`：变长字符串
- `NUMERIC(p, s)`：定点数，p 总位数、s 小数位
- `BOOLEAN`：布尔
- `TIMESTAMPTZ`：带时区时间戳
- `DATE`：日期（无时间）
- `JSONB`：二进制 JSON
- `ENUM`：PostgreSQL 枚举类型
- `TEXT[]`：文本数组

**通用字段**（每张表都有，下文表内不再重复说明）
- `<表名单数>_id`：主键，BIGINT NOT NULL，自增
- `created_at`：创建时间，TIMESTAMPTZ NOT NULL DEFAULT now()，不可变
- `updated_at`：更新时间，TIMESTAMPTZ NOT NULL DEFAULT now()，由触发器或 ORM 自动维护

**外键策略**
- 默认 `ON DELETE RESTRICT`（防止误删导致历史留痕悬空）
- 留痕表（`order_status_log`、`ingredient_stock_log`、`bot_conversation_log`）的外键也用 `RESTRICT`——即便关联业务记录被删，留痕必须保留可追溯
- 例外：子表（如 `customer_phone` → `customer`）用 `ON DELETE CASCADE`（父删则子删，避免孤儿数据）

#### 1.2.2 原料模块

##### `ingredient`（原料档案 + 当前库存）

需求文档第五章 §2（档案字段）与 §3（库存字段）合并为一张表——库存当前值与档案 1:1，独立成表徒增 join。

| 列名 | 类型 | 可空 | 默认 | 约束 / 说明 |
|---|---|---|---|---|
| ingredient_id | BIGINT | NO | - | 主键 |
| name | TEXT | NO | - | 原料名，UNIQUE；需求文档第五章 §2-(1) |
| unit | TEXT | NO | - | 计量单位（克/毫升/个/袋）；采购与配方共用 |
| unit_price | NUMERIC(10, 2) | YES | NULL | 最近采购平均单价；无采购记录时为空 |
| safety_stock | NUMERIC(10, 2) | NO | 0 | 安全库存线；低于此值提醒补货，0 = 不提醒 |
| shelf_life_days | INT | YES | NULL | 原料保质期（天）；**仅作记录**，原料不按批次管理、不计算临期（需求文档第五章 §3 末段） |
| storage_location | TEXT | YES | NULL | 存放位置（货架号、冰柜号） |
| common_source | TEXT | YES | NULL | 常用购买来源 |
| current_stock | NUMERIC(10, 2) | NO | 0 | 当前库存；**允许负数**（负数绝对值 = 欠缺量） |
| last_restocked_at | TIMESTAMPTZ | YES | NULL | 最近入库时间 |
| created_at | TIMESTAMPTZ | NO | now() | |
| updated_at | TIMESTAMPTZ | NO | now() | |

**索引**：UNIQUE(name)；BTREE(safety_stock)（低库存预警查询用：WHERE current_stock < safety_stock）

##### `ingredient_stock_log`（库存变动留痕）

| 列名 | 类型 | 可空 | 默认 | 约束 / 说明 |
|---|---|---|---|---|
| log_id | BIGINT | NO | - | 主键 |
| ingredient_id | BIGINT | NO | - | FK → ingredient；ON DELETE RESTRICT |
| change_amount | NUMERIC(10, 2) | NO | - | 本次变动量（正入库 / 负扣减）；CHECK ≠ 0 |
| new_balance | NUMERIC(10, 2) | NO | - | 变动后余额；与 ingredient.current_stock 同步 |
| source_type | STOCK_SOURCE | NO | - | ENUM：PURCHASE / FINISHED_GOODS_DEDUCT / LOSS / INVENTORY_ADJUSTMENT |
| reason | TEXT | NO | - | 必填原因（需求文档第五章 §4 要求） |
| order_id | BIGINT | YES | NULL | FK → `order`；source_type = FINISHED_GOODS_DEDUCT 时关联订单 |
| purchase_id | BIGINT | YES | NULL | FK → purchase；source_type = PURCHASE 时关联采购 |
| operator_user_id | BIGINT | NO | - | FK → user；操作人留痕 |
| created_at | TIMESTAMPTZ | NO | now() | |

**索引**：BTREE(ingredient_id, created_at)；BTREE(source_type)

**约束**：
- CHECK (change_amount <> 0)
- CHECK (
    (source_type = 'FINISHED_GOODS_DEDUCT' AND order_id IS NOT NULL) OR
    (source_type = 'PURCHASE' AND purchase_id IS NOT NULL) OR
    source_type IN ('LOSS', 'INVENTORY_ADJUSTMENT')
  )

**注意**：按技术方案 §4.2，**订单取消不写本表**——原料消耗在完工时已扣且不可回滚，取消订单时原料账面不再调整。

#### 1.2.5 订单模块

##### `order`（订单主表）

| 列名 | 类型 | 可空 | 默认 | 约束 / 说明 |
|---|---|---|---|---|
| order_id | BIGINT | NO | - | 主键；内部 ID（需求文档第二章 §3-(2)） |
| order_date | DATE | NO | - | 业务日期；用于编号生成与跨年查询 |
| daily_seq | INT | NO | - | 当日流水；从 1 起，参与编号生成 |
| customer_id | BIGINT | NO | - | FK → customer |
| menu_id | BIGINT | NO | - | FK → menu；引用商品必须在售（is_active = TRUE） |
| quantity | INT | NO | - | 数量；CHECK > 0 |
| pickup_at | TIMESTAMPTZ | NO | - | 取货/送达时间 |
| pickup_method | PICKUP_METHOD | NO | - | ENUM：PICKUP / DELIVERY |
| contact_phone | TEXT | NO | - | 本次联系电话；不关联 customer_phone，本次可临时改 |
| delivery_address | TEXT | YES | NULL | 本次配送地址；**pickup_method = DELIVERY 时必填**，自取时为 NULL |
| order_note | TEXT | YES | NULL | 本次订单备注；与 customer_note 独立维护 |
| payment_status | PAYMENT_STATUS | NO | 'UNPAID' | ENUM：UNPAID / PAID |
| payment_method | PAYMENT_METHOD | YES | NULL | ENUM：WECHAT / CASH / OTHER；**payment_status = PAID 时必填** |
| status | ORDER_STATUS | NO | 'PENDING' | ENUM：PENDING / COOKING / READY_FOR_PICKUP / READY_FOR_DELIVERY / DELIVERING / COMPLETED / CANCELLED |
| price | NUMERIC(10, 2) | NO | - | 订单总金额；等于 menu.price × quantity（应用层或生成列） |
| recipe_snapshot_id | BIGINT | YES | NULL | FK → recipe_snapshot；**status 进入 COOKING 后填**（技术方案 §4.2） |
| operator_user_id | BIGINT | YES | NULL | FK → user；浏览器端创建订单时填；机器人下单时为 NULL（操作人由 operator_wxid 在 bot_conversation_log 留痕替代） |
| created_at | TIMESTAMPTZ | NO | now() | 下单时间（精确到秒） |
| updated_at | TIMESTAMPTZ | NO | now() | |

**索引**：
- UNIQUE(order_date, daily_seq)：编号生成的并发安全（技术方案 §4.4）
- BTREE(status)：后厨屏按状态过滤
- BTREE(customer_id, created_at)：补录重复检测
- BTREE(pickup_at, status)：后厨今日看板按取货时间排序

**编号生成**（display_no）：**不存字段**，由 SQL 表达式或 ORM property 派生——`to_char(order_date, 'MMDD') || '-' || lpad(daily_seq::TEXT, 3, '0')`，例如 `0629-001`。单一可信源（display_no 是 order_date + daily_seq 的函数，存了易不一致）。

**约束**：
- CHECK (quantity > 0)
- CHECK (
    (pickup_method = 'DELIVERY' AND delivery_address IS NOT NULL) OR
    pickup_method = 'PICKUP'
  )
- CHECK (
    (payment_status = 'PAID' AND payment_method IS NOT NULL) OR
    payment_status = 'UNPAID'
  )

##### `order_status_log`（订单状态变更留痕）

| 列名 | 类型 | 可空 | 默认 | 约束 / 说明 |
|---|---|---|---|---|
| log_id | BIGINT | NO | - | 主键 |
| order_id | BIGINT | NO | - | FK → `order`；ON DELETE RESTRICT |
| old_status | ORDER_STATUS | YES | NULL | 旧状态；首次创建订单时为 NULL |
| new_status | ORDER_STATUS | NO | - | 新状态 |
| action | ORDER_ACTION | NO | - | ENUM：CREATE / UPDATE / START_COOKING / FINISH / MARK_DISPATCHED / CLOSE_PICKUP / CLOSE_DELIVERY / CANCEL |
| change_diff | JSONB | YES | NULL | 改单前后字段级 diff；仅 action = UPDATE 时填，结构见 §1.3.1 |
| note | TEXT | YES | NULL | 自由文本说明（取消原因等） |
| operator_user_id | BIGINT | YES | NULL | FK → user；浏览器端操作时填 |
| operator_wxid | TEXT | YES | NULL | 机器人端操作时填；与 operator_user_id 二选一 |
| created_at | TIMESTAMPTZ | NO | now() | |

**索引**：BTREE(order_id, created_at)

  (operator_user_id IS NOT NULL AND operator_wxid IS NULL) OR
  (operator_user_id IS NULL AND operator_wxid IS NOT NULL)

**注意**：所有状态变更必须经 service 层事务（更新 `order.status` + 写本表 + 触发库存联动 + WebSocket 推送），禁止直接改 `status`（技术方案 §4.3）。`change_diff` 与 `note` 分工：前者记结构化字段变更，后者记自由文本。

#### 1.2.6 采购模块

#### 1.2.7 鉴权模块

##### `user`（PIN 登录用户）

| 列名 | 类型 | 可空 | 默认 | 约束 / 说明 |
|---|---|---|---|---|
| user_id | BIGINT | NO | - | 主键 |
| pin_hash | TEXT | NO | - | bcrypt 哈希；**不加 UNIQUE 不加索引**（bcrypt 同明文不同 hash，无法 SELECT 匹配，登录靠遍历 checkpw，详见技术方案 §4.2） |
| display_name | TEXT | NO | - | 显示名（如「店主」「店员张三」「后厨李四」）；写入所有留痕表的「操作人」字段 |

| enabled | BOOLEAN | NO | TRUE | 账号启用开关 |
| failed_attempts | INT | NO | 0 | PIN 错误次数；累计 5 触发锁定 |
| locked_until | TIMESTAMPTZ | YES | NULL | 锁定截止时间；NULL = 未锁 |
| last_login_at | TIMESTAMPTZ | YES | NULL | 最近登录时间 |
| created_at | TIMESTAMPTZ | NO | now() | |
| updated_at | TIMESTAMPTZ | NO | now() | |

**索引**：BTREE(display_name)；部分索引 `WHERE enabled = TRUE`（登录遍历时只扫启用用户）

**约束**：CHECK (failed_attempts >= 0 AND failed_attempts <= 5)

##### `api_token`（程序 Bearer Token）

| 列名 | 类型 | 可空 | 默认 | 约束 / 说明 |
|---|---|---|---|---|
| token_id | BIGINT | NO | - | 主键 |
| token_hash | TEXT | NO | - | bcrypt 哈希；UNIQUE（与 pin_hash 不同——token 校验是先 hash 再 SELECT 比对，可加索引） |
| name | TEXT | NO | - | 凭证名称（如「OpenClaw 主机」） |
| enabled | BOOLEAN | NO | TRUE | 启用开关 |
| last_used_at | TIMESTAMPTZ | YES | NULL | 最近使用时间 |
| created_at | TIMESTAMPTZ | NO | now() | |

**索引**：UNIQUE(token_hash)

**与 `user` 表的区别**（技术方案 §4.2）：user 给人登录（有 PIN 锁定逻辑），api_token 给程序调用（无锁定逻辑、泄露靠轮换）；凭证类型不同、语义不应混淆。

##### `bot_conversation_log`（机器人对话留痕）

| 列名 | 类型 | 可空 | 默认 | 约束 / 说明 |
|---|---|---|---|---|
| log_id | BIGINT | NO | - | 主键 |
| session_id | TEXT | NO | - | 一次接单流程的多次交互聚合；由 OpenClaw Plugin 从 `ctx.sessionKey` 派生 |
| step_type | CONVERSATION_STEP | NO | - | ENUM：CUSTOMER_ORIGINAL_MSG / RECOGNITION_RESULT / BOT_REPLY / CUSTOMER_REPLY / ORDER_PERSISTED；对应需求文档第九章 §4-(5) 五类痕迹 |
| payload | JSONB | NO | - | 该步骤内容，按 step_type 不同结构不同，见 §1.3.3 |
| operator_wxid | TEXT | YES | NULL | 店员侧留痕来源；前四类填，第五类（订单入库）可不填（已有 order_id 关联） |
| order_id | BIGINT | YES | NULL | FK → `order`；仅 step_type = ORDER_PERSISTED 时填，回填关联订单 |
| created_at | TIMESTAMPTZ | NO | now() | |

**索引**：
- BTREE(session_id, created_at)：按接单会话追溯完整对话流
- BTREE(operator_wxid, created_at)：按店员查历史接单痕迹
- 部分索引 `WHERE order_id IS NOT NULL`：按订单查关联对话

**写入路径**（技术方案 §4.2 + §3.4）：
- 前四类：OpenClaw Plugin 注册 `message_received` / `message_sent` / `before_tool_call` hook，POST 到 `/api/internal/bot-conversation-log`
- 第五类：MCP 工具 `create_order` 在 service 层事务中直接写入（与订单入库同事务）

### 1.3 JSONB 字段结构

#### 1.3.1 `order_status_log.change_diff`

改单（action = UPDATE）前后的字段级 diff，**仅记变更字段**、未变字段不记：

```json
{
  "quantity": {"old": 1, "new": 2},
  "pickup_at": {
    "old": "2026-06-29T15:00:00+08:00",
    "new": "2026-06-29T16:00:00+08:00"
  },
  "order_note": {"old": null, "new": "加巧克力粉"}
}
```

**字段级 diff 的好处**：回执可以渲染「数量从 1 改为 2、取货时间延后 1 小时」；事后审计可精确追溯哪次改了哪个字段。

**与 `note` 字段分工**：`note` 记取消原因等自由文本，`change_diff` 记结构化字段变更，互不重叠。

#### 1.3.2 `recipe_snapshot.content`

订单进入「制作中」时锁定的完整配方内容：

```json
{
  "menu_name": "巧克力蛋糕",
  "steps": [
    {"step_no": 1, "description": "打发蛋白至硬性发泡", "duration_minutes": 10},
    {"step_no": 2, "description": "烘焙 180°C 30 分钟", "duration_minutes": 30}
  ],
  "total_time_minutes": 40,
  "loss_rate": 0.05,
  "allergens": ["乳制品", "麸质"],
  "ingredients": [
    {"ingredient_id": 12, "name": "低筋面粉", "quantity": 200, "unit": "克"},
    {"ingredient_id": 18, "name": "黑巧克力", "quantity": 150, "unit": "克"}
  ]
}
```

**为什么快照**：配方随时可改，但订单按制作那一刻的配方扣料、按当时的损耗率算成本；快照确保扣料口径不被后续配方修改污染。

#### 1.3.3 `bot_conversation_log.payload`

按 step_type 不同结构不同。JSONB 不强约束子结构（不同 step_type 不同结构），由应用层保证写入格式正确：

```json
// step_type = CUSTOMER_ORIGINAL_MSG（店员原始消息）
{"text": "小张订1个巧克力蛋糕，明天下午3点取，电话138xxxx"}

// step_type = RECOGNITION_RESULT（识别结果）
{
  "customer_wxid": "wxid_abc123",
  "menu_name": "巧克力蛋糕",
  "quantity": 1,
  "pickup_at": "2026-06-30T15:00:00+08:00",
  "contact_phone": "138xxxx",
  "missing_fields": []
}

// step_type = BOT_REPLY（机器人回执）
{"text": "已下单，订单号 0629-001"}

// step_type = CUSTOMER_REPLY（店员回复）
{"text": "是的，按这个"}

// step_type = ORDER_PERSISTED（最终入库订单）
{"order_id": 42, "order_no": "0629-001"}
```

#### 1.3.4 `recipe.steps`

配方制作步骤数组（同时作为 `recipe.steps` 字段结构与 `recipe_snapshot.content.steps` 元素结构）：

```json
[
  {"step_no": 1, "description": "打发蛋白至硬性发泡", "duration_minutes": 10},
  {"step_no": 2, "description": "烘焙 180°C 30 分钟", "duration_minutes": 30},
  {"step_no": 3, "description": "出炉冷却", "duration_minutes": 15}
]
```

每步含序号、描述、可选耗时（需求文档第六章 §2-(3)(4)）。`recipe.total_time_minutes` 可由各步 `duration_minutes` 求和得出，存了便于直接查询。

### 1.7 与技术方案 §4 / 需求文档的细化说明

本次落地过程中的几处细化（不冲突、仅补充）：

1. **`order.order_date + daily_seq` 拆分**：技术方案 §4.2 说「下单日期 + 当日编号联合唯一索引」，本设计拆为 DATE + INT 两个字段，便于编号生成与查询；display_no（如「0629-001」）不存字段、由两字段派生（单一可信源）
2. **`ingredient` 表合并原料档案与当前库存**：需求文档第五章 §2 是档案字段、§3 是库存字段，技术方案 §4.2 也是合并设计，本设计沿用——`current_stock` 是当前值、`ingredient_stock_log` 是变动留痕，二者配合
3. **`bot_conversation_log.payload` 用 JSONB 不分子表**：5 类痕迹结构差异大、JSONB 灵活度高；如未来查询性能成问题再考虑拆表
4. **`order_status_log.action` 用枚举**：技术方案 §4.2 只说「触发动作」，本设计细化为 8 种枚举（CREATE/UPDATE/START_COOKING/FINISH/MARK_DISPATCHED/CLOSE_PICKUP/CLOSE_DELIVERY/CANCEL），便于按动作类型筛选与统计
5. **`api_token.token_hash` 加 UNIQUE 索引、`user.pin_hash` 不加**：技术方案 §4.2 已说明差异，本设计在索引层落地——token 校验是「客户端传明文 → 服务端 bcrypt.hash → SELECT 比对」可走索引，PIN 校验是「遍历 enabled 用户逐个 checkpw」无法走索引
6. **留痕表外键统一 `ON DELETE RESTRICT`**：技术方案未明确，本设计补充——即便关联业务记录被删，留痕必须保留可追溯
